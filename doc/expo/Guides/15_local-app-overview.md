# Build locally: Overview

## 文档解决的问题

这篇文档是 Expo 本地构建总览页，解释为什么你可能需要在自己的机器上构建 App、本地构建和 EAS Build 的关系是什么，以及本地调试构建、生产构建、构建缓存、Android 预编译模块分别指什么。

## 适用场景

- 你要在本机快速调试原生代码或平台差异。
- 你所在环境对网络访问有限，不能完全依赖云构建。
- 你想自己掌控签名凭证、构建缓存或 Android 模块源码编译。

## 核心概念

### 本地构建不是替代 EAS，而是补充

文档明确说明，本地构建与 EAS Build 并不冲突。你可以继续使用云端自动化，同时在需要时回到本机构建。

### Debug build 与 Release build

- Debug build：主要用于开发、调试、快速反馈。
- Release build：用于上架或接近上架的正式产物。

### 本地构建的前提

你需要本机具备原生开发环境：

- Android Studio
- Xcode

## 关键流程

### 1. 准备本地原生环境

文档把这列为明确前置条件。

### 2. 本地创建调试构建

使用：

```sh
npx expo run:android
npx expo run:ios
```

这会调用本机 Android SDK / Xcode 构建调试版应用。

### 3. 本地创建发布构建

文档说明，发布构建需要你自己管理签名凭证，并手动生成上架包，再手动提交到 Google Play / App Store。

### 4. 可选：复用构建缓存

你可以使用 EAS 作为构建缓存提供方，也可以自己做缓存提供方。

### 5. 可选：控制 Android 预编译 Expo Modules

文档说明 Android 端提供预编译模块，可减少 Gradle 工作量；你也可以在需要修改模块源码时选择退出这套默认行为。

## 注意事项、限制条件和坑点

- 本地构建要求你自己装好并维护 Android Studio / Xcode 环境。
- 发布构建意味着你要自己处理签名和手动提交流程，工作量高于调试构建。
- 某些场景必须本地构建，例如网络受限环境、需要本地凭证管理、需要定制缓存或源码级模块调试时。

## React Web 开发者容易误解的点

- 在 Web 项目里，本地跑和发布打包通常都靠同一套前端工具链；移动端本地构建还依赖 Android / iOS 原生编译工具。
- “本地构建”不只是 `npm run build` 的概念，而是生成原生二进制。
- 云构建不是唯一正解，本地构建在原生调试阶段往往更高效。

## 实际开发建议

- 如果你在写原生代码或排查平台差异，优先用本地调试构建。
- 如果团队已经接入 EAS Build，也不要放弃本地构建能力，它对问题定位很重要。
- 如果只是做日常 JS 业务开发，可按团队情况继续使用云构建和远端自动化。
- 基于文档内容推导：本地构建能力更像移动端团队的基础设施，不应该完全外包给云服务。

## 文档明确说明

- 本地构建适用于调试原生代码、网络受限环境、自管凭证、接入缓存、退出 Android 预编译模块等场景。
- Debug build 可用 `npx expo run:[android|ios]` 创建。
- Release build 需要本地生成签名产物并手动提交。
- 本地构建是 EAS Build 的补充，而不是对立面。

## 基于文档内容推导

- 团队如果完全不会本地构建，遇到原生问题时排查成本会很高。
- 本地构建知识应尽早补齐，哪怕日常主要使用云构建。
- 当前文档未涉及具体 Android / iOS 环境安装步骤，也未展开发布构建签名细节，只提供导航入口。
