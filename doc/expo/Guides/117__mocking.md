# 在 Expo 模块中模拟原生调用（Mocking Native Calls）

本指南介绍如何在 Expo 模块中模拟（mock）原生函数。模拟是指在单元测试中用"不执行任何实际操作的假版本"来替换真实函数的技术。

> **关键术语解释（面向初学者）：**
> - **Mock（模拟/打桩）**：在测试中，用一个可控的假实现替换真实的函数或模块，使得测试可以不依赖真实环境而独立运行。
> - **Native Module（原生模块）**：用 Swift/Kotlin 等原生语言编写的、运行在 iOS/Android 设备上的代码。JavaScript 无法直接在普通计算机上执行这些代码。
> - **Jest**：JavaScript 生态中最流行的测试框架，提供断言（assertion）、模拟（mocking）等测试功能。
> - **jest-expo**：Expo 官方提供的 Jest 预设配置（preset），为 Expo 项目预配置好了适合的测试环境。
> - **Unit Test（单元测试）**：针对最小可测试单元（通常是函数或组件）进行的自动化测试。

Expo 推荐使用 **Jest** 配合 **`jest-expo`** 预设来进行单元测试。由于测试依赖原生代码的应用需要真实设备，而开发者通常希望在本地计算机上运行测试，因此需要用模拟来替换真实逻辑。

Expo SDK 已经为其生态中的社区库提供了默认的模拟实现。同时，开发者也可以使用 Jest 自带的 API 手动模拟 JavaScript 代码。此外，模块作者可以将默认的模拟实现直接打包在自己的模块中，这样使用者在运行测试时会自动获取这些模拟——无需额外配置。

---

## 为模块提供模拟实现（Providing Mocks for a Module）

模块作者可以为自己的原生模块提供默认的模拟实现。具体做法是：

1. 在模块项目中创建一个名为 `__mocks__` 的目录
2. 在该目录下创建一个与原生模块同名的文件
3. 从该文件中导出模拟实现

当使用 `jest-expo` 预设运行测试时，测试框架会通过 `requireNativeModule` 调用自动返回这些导出的模拟实现，而非真实的原生模块。

> **关键术语解释：**
> - **`__mocks__` 目录**：Jest 约定俗成的模拟文件夹名称。Jest 在运行时会自动查找此目录中的文件来替换同名模块。
> - **`requireNativeModule`**：Expo 模块系统中用于获取原生模块引用的函数。在测试环境下，`jest-expo` 预设会拦截此调用并返回 `__mocks__` 目录中的模拟实现。

### 示例：模拟 `expo-clipboard`

以 `expo-clipboard` 为例，需要在 `__mocks__` 目录下创建一个名为 `ExpoClipboard.ts` 的文件：

**文件路径：** `__mocks__/ExpoClipboard.ts`

```ts
export async function hasStringAsync(): Promise<boolean> {
  return false;
}
```

创建此文件后，在测试中调用 `hasStringAsync()` 时将返回 `false`，而不会真正去查询设备剪贴板。

## 自动生成模拟实现（Automatic Generation of Mocks）

对于包含大量方法的模块，手动维护模拟实现非常繁琐。为此，Expo 提供了一个自动化脚本来根据 Swift 代码生成 TypeScript 或 JavaScript 格式的模拟文件。

> **注意/限制：**
> - 自动生成工具仅解析 **Swift（iOS）** 代码来生成模拟。
> - **Android 专属 API**（例如仅存在于 Kotlin 中的方法）**不会被自动生成**，需要开发者手动添加或修改对应的模拟。
> - 该工具依赖 **SourceKitten** 框架来解析 Swift 源码，需要先安装此依赖。

### 安装与使用

首先安装 SourceKitten：

```sh
brew install sourcekitten
```

然后在包含 `expo-module.config.json` 配置文件的模块根目录下运行生成命令：

```sh
npx expo-modules-test-core generate-ts-mocks
```

执行后，会在 `__mocks__` 目录下生成一个 TypeScript 文件，其中包含每个原生方法和原生视图的模拟实现。

> **提示：** 如果需要生成 JavaScript 格式的模拟文件（而非 TypeScript），可以运行 `generate-js-mocks` 命令：
>
> ```sh
> npx expo-modules-test-core generate-js-mocks
> ```

> **基于经验建议：**
> - 建议将自动生成的模拟文件纳入版本控制（commit 到 Git），这样 CI/CD 环境中的测试无需每次重新生成。
> - 如果原生模块的 API 发生了变更，记得重新运行生成命令来更新模拟文件。
> - 对于仅 Android 平台存在的方法，建议在生成的模拟文件中手动添加对应的空实现或默认返回值，以确保跨平台测试的完整性。

## 使用模拟模块进行单元测试（Unit Testing with Mocked Modules）

创建好模拟实现后，就可以编写测试来验证 JavaScript 代码是否正确调用了原生方法并正确处理了返回结果。

使用上述 `generate-ts-mocks` 命令生成的模拟文件大致结构如下（以示例模块为例）：

**文件路径：** `example-module/__mocks__/ExpoModuleName.ts`

```ts
/**
 * Automatically generated by expo-modules-test-core.
 *
 * This autogenerated file provides a mock for native Expo module,
 * and works out of the box with the expo jest preset.
 *
 */

export type URL = any;

export function hello(): any {}

export async function setValueAsync(value: string): Promise<any> {}

export type ViewProps = {
  url: URL;
  onLoad: (event: any) => void;
};

export function View(props: ViewProps) {}
```

> **关键术语解释：**
> - **`jest.fn()`**：Jest 提供的模拟函数创建工具，可以追踪函数的调用次数、参数等信息。
> - **`toHaveBeenCalledWith()`**：Jest 断言方法，用于验证某个函数是否以特定参数被调用过。
> - **`toHaveBeenCalledTimes()`**：Jest 断言方法，用于验证某个函数被调用的次数。
> - **`renderHook()`**：来自 `@testing-library/react-native` 的工具函数，专门用于在隔离环境中测试 React Hook。

以下各节展示了使用 `expo-clipboard`、`expo-screen-capture` 和 `expo-app-integrity` 等真实 SDK 模块的实际测试模式。

### 基础测试配置（Basic Test Setup）

将测试文件放在源码旁边的 `__tests__` 目录中。在测试中同时导入主模块和原生模块的模拟实现：

```js
import * as MyModule from '../MyModule';
import ExpoMyModule from '../ExpoMyModule';

describe('MyModule', () => {
  it('calls native module with correct parameters', async () => {
    await MyModule.doSomething('test-param');
    expect(ExpoMyModule.doSomething).toHaveBeenCalledWith('test-param');
  });
});
```

> **代码解析：**
> - `MyModule` 是面向 JavaScript 开发者的公开 API 模块。
> - `ExpoMyModule` 是对原生模块的直接引用（在测试环境下被替换为模拟实现）。
> - 测试验证的是：调用 `MyModule.doSomething('test-param')` 时，底层是否正确将 `'test-param'` 参数转发给了原生模块。

### 测试函数调用与返回值（Testing Function Calls and Return Values）

利用 Jest 的断言方法来验证 JavaScript 函数是否正确委托给原生实现：

```js
describe('Module functionality', () => {
  it('delegates to native implementation', () => {
    MyModule.setData('test-data');
    expect(ExpoMyModule.setDataAsync).toHaveBeenCalledWith('test-data', {});
  });

  it('handles async operations', async () => {
    await expect(MyModule.getDataAsync()).resolves.not.toThrow();
  });

  it('verifies call count', () => {
    MyModule.performAction();
    MyModule.performAction();
    expect(ExpoMyModule.performAction).toHaveBeenCalledTimes(2);
  });
});
```

> **代码解析：**
> - 第一个测试验证同步调用 `setData` 是否正确转发为原生的 `setDataAsync` 方法调用，并附带正确的参数。
> - 第二个测试验证异步操作 `getDataAsync` 不会抛出异常（即正常完成）。
> - 第三个测试验证调用计数——调用两次 `performAction` 后，原生模块的对应方法也应该被调用了恰好两次。

### 测试与原生模块交互的 React Hook（Testing React Hooks with Native Modules）

对于使用了原生模块的 React Hook，需要使用 `@testing-library/react-native` 提供的 `renderHook` 工具来进行测试：

```js
import { renderHook } from '@testing-library/react-native';
import { useMyHook } from '../useMyHook';
import ExpoMyModule from '../ExpoMyModule';

jest.mock('../ExpoMyModule', () => ({
  startOperation: jest.fn().mockResolvedValue(),
  stopOperation: jest.fn().mockResolvedValue(),
}));

describe('useMyHook', () => {
  it('calls native methods on mount and unmount', () => {
    const hook = renderHook(useMyHook);
    expect(ExpoMyModule.startOperation).toHaveBeenCalledTimes(1);

    hook.unmount();
    expect(ExpoMyModule.stopOperation).toHaveBeenCalledTimes(1);
  });

  it('handles parameter changes', () => {
    const hook = renderHook(useMyHook, { initialProps: 'param1' });

    hook.rerender('param2');

    expect(ExpoMyModule.startOperation).toHaveBeenCalledTimes(2);
    expect(ExpoMyModule.stopOperation).toHaveBeenCalledTimes(1);
  });
});
```

> **代码解析：**
> - `jest.mock()` 用于手动定义模拟实现，这里将 `startOperation` 和 `stopOperation` 都定义为返回已解决 Promise 的模拟函数。
> - **挂载测试**：Hook 挂载（mount）时应调用 `startOperation` 一次，卸载（unmount）时应调用 `stopOperation` 一次。
> - **参数变更测试**：当参数从 `'param1'` 变为 `'param2'`（通过 `rerender`）时，`startOperation` 应被调用两次（初始一次 + 参数变更一次），`stopOperation` 应被调用一次（参数变更时先停止旧操作）。

> **基于经验建议：**
> - 在测试 Hook 时，`renderHook` 比直接渲染完整组件更高效，因为它只关注 Hook 本身的逻辑，不涉及 UI 渲染。
> - 使用 `jest.mock()` 手动定义模拟时，确保模拟函数的返回类型与真实函数一致（例如异步函数需要返回 Promise），否则测试行为可能与实际运行不符。

## 最佳实践（Best Practices）

- **每次测试后清理状态：** 使用 `beforeEach` 或 `afterEach` 生命周期钩子重置模拟（例如调用 `jest.clearAllMocks()`），防止测试之间的状态污染。
- **测试边界情况：** 验证原生调用失败或返回异常数据时，系统的反应是否符合预期。使用 `mockRejectedValue` 模拟错误场景。
- **使用明确的测试名称：** 测试描述应准确说明正在验证的具体行为，而非笼统的"测试某功能"。
- **分组相关测试：** 使用 `describe` 代码块按功能或组件对测试进行逻辑分组，提高可读性和维护性。

> **基于经验建议：**
> - 在 `beforeEach` 中调用 `jest.clearAllMocks()` 是一个好习惯，它比 `jest.resetAllMocks()` 更温和——只清除调用记录而不重置模拟实现本身。
> - 对于复杂模块，建议为每个公开方法编写独立的 `describe` 块，每个块内包含正常路径、异常路径和边界条件的测试用例。
> - 模拟的返回值应尽量反映真实场景的数据结构，过于简化的返回值（如始终返回 `null`）可能掩盖实际的序列化/反序列化问题。
