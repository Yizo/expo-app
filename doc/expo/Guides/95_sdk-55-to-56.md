# Expo Router 从 SDK 55 迁移到 SDK 56

## 文档解决的问题

SDK 56 起，Expo Router 不再允许应用代码直接从外部 `@react-navigation/*` 包导入。迁移只需要把模块来源改到对应的 `expo-router` 入口；运行时 API 本身没有改变。

本文适合已经使用 Expo Router、准备从 SDK 55 升级到 SDK 56 的项目。它不是“从 React Navigation 架构迁移到 Expo Router 文件路由”的完整指南。

## 自动迁移

在项目根目录运行 codemod：

```sh
npx expo-codemod sdk-56-expo-router-react-navigation-replace src
```

把 `src` 替换为应用代码所在目录或 glob。该工具会把应用代码中的 `@react-navigation/*` import 改写为匹配的 `expo-router` 入口。

其他包管理器：

```sh
yarn dlx expo-codemod sdk-56-expo-router-react-navigation-replace src
pnpm dlx expo-codemod sdk-56-expo-router-react-navigation-replace src
bunx expo-codemod sdk-56-expo-router-react-navigation-replace src
```

## 手动迁移

示例：

```tsx
// SDK 55
import { ThemeProvider, DarkTheme } from '@react-navigation/native';
import { createMaterialTopTabNavigator } from '@react-navigation/material-top-tabs';

// SDK 56
import {
  ThemeProvider,
  DarkTheme,
} from 'expo-router/react-navigation';
import { createMaterialTopTabNavigator } from 'expo-router/js-top-tabs';
```

完整映射：

| SDK 55 来源 | SDK 56 目标 |
| --- | --- |
| `@react-navigation/native` | `expo-router/react-navigation` |
| `@react-navigation/core` | `expo-router/react-navigation` |
| `@react-navigation/elements` | `expo-router/react-navigation` |
| `@react-navigation/routers` | `expo-router/react-navigation` |
| `@react-navigation/stack` | `expo-router/js-stack` |
| `@react-navigation/bottom-tabs` | `expo-router/js-tabs` |
| `@react-navigation/material-top-tabs` | `expo-router/js-top-tabs` |
| `@react-navigation/native-stack` | 无直接等价入口，改用 Expo Router `Stack` 布局。 |
| `@react-navigation/drawer` | 无直接等价入口，改用 Expo Router `Drawer` 布局。 |

## 第三方库兼容

许多第三方库仍会从 `@react-navigation/core` 导入。为降低 SDK 56 迁移成本，Expo CLI 会对来自 `node_modules` 的这些 import 自动重写为 Expo Router 入口。

这个兼容层只处理第三方依赖，不会自动放过应用代码中的旧 import。它也是临时措施，未来会在提供专门的库迁移指南后移除。

## 关闭检查与兼容重写

启动 bundler 前设置：

```sh
EXPO_ROUTER_DISABLE_RN_NAVIGATION_CHECK=1 npx expo start
```

这会同时：

- 禁用针对应用代码 `@react-navigation/*` import 的 bundler 错误。
- 禁用 Expo CLI 对 `node_modules` 中相关 import 的自动兼容重写。

因此它是显式退出迁移检查和兼容层的开关，不是推荐的长期迁移方案。

## 注意事项与坑点

- 变化发生在模块 specifier，不是 API 语义；不要在迁移时无理由重写业务逻辑。
- Codemod 的目标参数必须覆盖真正的应用代码目录，否则会遗漏旧 import。
- `native-stack` 和 `drawer` 没有简单的一对一模块替换，需要采用 Expo Router 布局。
- 第三方库暂时能运行，不代表其旧 import 已永久受支持。
- 环境变量会关闭检查及第三方兼容重写，可能让问题延后暴露。

## React Web 开发者容易误解的地方

- 这类似包入口重组，而不是 React 组件 API 大升级：导入路径变化，实际调用方式通常不变。
- `node_modules` 自动改写只是一层 bundler 兼容，不会修改依赖包磁盘源码，也不会替应用代码迁移。
- Expo Router 的 `Stack`/`Drawer` 布局由文件系统路由驱动，不能简单把旧 Navigator factory 的 import 字符串替换掉就结束。

## 实际开发建议

> **基于文档内容推导：** 先运行 codemod，再搜索应用代码中残留的 `@react-navigation/` import，并逐项按映射表处理；对 `native-stack` 和 `drawer` 单独做导航结构验证。

> **基于文档内容推导：** 只有在定位第三方兼容问题时才临时使用 `EXPO_ROUTER_DISABLE_RN_NAVIGATION_CHECK=1`，并记录原因；正常迁移完成后应恢复默认检查。

## 文档明确说明与未涉及内容

文档明确说明 SDK 56 的核心变化仅为导入来源迁移，并提供 codemod、手动映射、第三方兼容层和退出环境变量。

当前文档未涉及：Expo SDK 本身的完整升级步骤、依赖版本检查命令、原生工程升级、所有第三方导航库的兼容状态、兼容层移除的具体版本。
