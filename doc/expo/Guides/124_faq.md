# Expo 推送通知故障排查与常见问题

> 对应原文：<https://docs.expo.dev/push-notifications/faq.md>

## Expo Push Service 常见问题

### 是否收费

通过 Expo Push Service 发送通知不收费。

### 发送速率限制

每个项目最多每秒发送 600 条通知。超过后，后续请求会失败，直到速率降回每秒 600 条以下。

服务端应实现节流和重试。`expo-server-sdk-node` 已自动处理节流。

### 是否必须使用 Expo Push Service

不是。Expo 项目可以使用其他推送服务。通过 `getDevicePushTokenAsync` 获取原生设备 token 后，可交给其他服务，也可以直接连接 FCM/APNs。

### 连接与内容存储

- Expo 到 Apple 和 Google 的连接使用 HTTPS 加密。
- 通知内容只在交付所需时间内保存在内存和消息队列，不存入数据库。
- Expo 团队主动调试服务时，可能在断点等调试场景看到通知内容；除此之外 Expo 无法查看通知内容。

### 投递保证

Expo 尽力把消息送达 Google/Apple，并以“至少一次”方式设计底层服务交付。少数情况下消息可能重复或完全未交付。

push receipt 只记录消息是否成功交给底层推送服务。Google 和 Apple 再按各自策略投递到设备，因此 receipt 并不等于最终设备送达。

### `ExpoPushToken` 何时变化

- 应用升级时 token 保持不变。
- Android 重装应用可能改变 token。
- iOS 即使卸载重装，token 仍保持不变。
- 修改 `applicationId` 或 `experienceId` 会改变 token。
- `ExpoPushToken` 本身不会过期，但用户卸载后，Expo 可能返回 `DeviceNotRegistered`；此时应停止向该 token 发送。

## 通知完全不工作

推送链路包含多个环节，应先检查 push ticket 和 push receipt 的错误信息。

还可以先测试本地通知。如果本地通知和客户端逻辑正常，问题范围就更可能在服务器发送或应用凭据。

发送测试通知：

```sh
curl -H "Content-Type: application/json" \
  -X POST "https://exp.host/--/api/v2/push/send" \
  -d '{
    "to": "ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]",
    "title": "hello",
    "body": "world"
  }'
```

用返回的 ticket `id` 查询回执：

```sh
curl -H "Content-Type: application/json" \
  -X POST "https://exp.host/--/api/v2/push/getReceipts" \
  -d '{
    "ids": ["XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX"]
  }'
```

## 开发环境正常，release 不工作

这通常表示生产应用没有配置凭据或凭据配置错误。

从 SDK 53 起，Expo Go 不支持推送通知，应使用 development build 测试。SDK 52 及更早版本的 Expo Go 使用 Expo 自己的凭据，因此过去可能在未配置自有凭据时也能在开发环境工作；这种旧体验不能证明生产凭据正确。

发布到应用商店时必须使用自己的凭据：

- Android 按 FCM V1 指南配置。
- iOS 使用 push key；若 key 被撤销，通知会失败，可通过 `eas credentials` 添加新 key。

## Android 偶尔收不到通知

可能与发送时的 `priority` 有关。Expo 接受：

- `default`：映射为 Apple/Google 的默认优先级。
- `high`：映射为平台高优先级。
- `normal`：映射为平台普通优先级。
- 省略字段：与 `default` 完全相同。

设置为 `high` 能提高 Android 显示通知的可能性，但文档没有承诺必达。

## 推送凭据过期

iOS 凭据过期时运行：

```sh
eas credentials
```

选择 iOS 和对应 build profile，移除旧 push notification key，再生成新的 key。

## iOS：`No valid aps-environment entitlement string found`

该错误表示 iOS 项目没有设置 push notification key。可以先在项目 Credentials 页面检查，再触发新构建：

```sh
eas build --profile [profile] --platform ios
```

## 发送时出现错误

检查返回 ticket 或 receipt 的 `details` 属性，其中会包含更具体的信息。不要只看顶层 HTTP 状态或 `message`。

## iOS 获取 push token 很慢

`getDevicePushTokenAsync` 和 `getExpoPushTokenAsync` 在 iOS 上有时需要很久才能完成，这超出 `expo-notifications` 的控制范围。设备可能没有网络、处于飞行模式，或 APNs 注册链路暂时不可用；应用不应因此整体停止运行，只需暂时禁用依赖推送的功能。

原文汇总的排查方向包括：

- 确保设备网络可靠，尝试关闭 Wi-Fi或切换网络，并检查防火墙是否阻止 5223 端口。
- Bare React Native 应用需要手工启用 Push Notifications capability。
- APNs 区域服务可能暂时异常，可稍后甚至数日后重试。
- 关闭设备网络共享。
- 若同一设备从 Xcode 构建切换到 TestFlight 构建，可能需要重启设备。
- 未配置 SIM 卡的设备可尝试完成 SIM 设置。

这些是社区成员曾采用的解决方式，不是每种情况都必然有效。

## Android 通知图标变成灰色或白色方块

这是通知图标资源不符合 Android 要求。图片应为全白图形并使用透明背景；该限制由 Google 强制，不是 Expo 自行规定。

## 推荐排障顺序

**基于文档内容推导：** 可以按以下顺序缩小范围：

1. 用本地通知验证客户端展示和监听逻辑。
2. 确认正在使用 development build，而不是 SDK 53+ 的 Expo Go。
3. 检查 push token 是否与当前 `applicationId`/项目匹配。
4. 检查 ticket，再检查 receipt 的 `details`。
5. 核对 Android FCM 或 iOS push key/entitlement。
6. 最后排查设备网络、系统设置和平台服务波动。
