# Expo 推送通知服务概览

> 对应原文：<https://docs.expo.dev/push-notifications/overview.md>

## 文档定位

这是一篇导航性质的概览，说明 Expo 推送通知服务解决什么问题，以及后续应按什么主题继续学习。它不提供完整代码或配置步骤。

推送通知在移动端需要与两个平台服务通信：

- Android 使用 Firebase Cloud Messaging（FCM）。
- iOS 使用 Apple Push Notification Service（APNs）。

Expo 负责处理其中大量平台通信复杂度，让应用前端和后端可以用更统一的方式对待 Android 与 iOS 通知，从而减少分别实现两套流程的时间。

对 React Web 开发者来说，可以把 Expo Push Service 理解为位于业务服务器与 FCM/APNs 之间的一层统一网关；浏览器 Web Push 的 Service Worker 模型并不是本页讨论对象。

## 推荐学习路径

原文把完整任务拆成以下主题：

1. **先理解通知类型和行为**：了解不同通知类型，以及应用处于前台、后台或关闭状态时的差异。
2. **设置推送通知**：配置项目，获取推送 token 和平台凭据。
3. **发送通知**：由服务器调用 Expo Push Service API。
4. **处理收到的通知**：应用监听通知到达或用户点击事件，并执行相应操作。
5. **排查问题**：查阅常见问题和故障排查说明。

原文还提供一段完整视频，覆盖 Android FCM v1、Android/iOS EAS 凭据、EAS Build 和 Expo Notifications 测试工具。

## 容易误解的地方

- Expo 简化了 FCM/APNs 的使用，但底层推送最终仍由 Google 和 Apple 的服务发送到设备。
- “统一处理 Android 与 iOS”不表示两个平台的通知行为完全一致；本页只说明 Expo 减少了通信层的差异。
- 客户端接收逻辑只是整个链路的一部分，服务端发送、token、凭据和事件处理都需要单独完成。

## 实际使用建议

**基于文档内容推导：** 初次接入时应按概览给出的顺序推进，不要从“发送 API”直接开始。先理解通知行为，再完成 token 与凭据配置，可以减少把权限、平台状态或凭据问题误判成服务端发送失败。
