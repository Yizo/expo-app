# 开发与生产环境

> **原始文档地址**：[https://docs.expo.dev/workflow/development-mode/](https://docs.expo.dev/workflow/development-mode/)

了解如何在开发（Development）和生产（Production）配置下启动和运行你的应用。

---

## 概述

Expo 应用始终运行在两种环境状态之一：

- **开发模式（Development Mode）**：通过标准启动命令（如 `npx expo start`）在本地运行时，默认进入开发模式。此模式提供有用的警告信息和调试工具，帮助开发者更高效地排查问题。
- **生产模式（Production Mode）**：已发布的项目或通过独立构建（Standalone Build）生成的应用将以生产模式运行。此模式会压缩（Minify）你的代码，并准确反映最终用户的实际体验。

下面我们将详细介绍这两种模式以及如何切换它们。

> **初学者提示**：
> - **开发模式** 类似于你在工厂车间里调试机器——所有仪表盘都打开，所有警告灯都亮着，方便你发现和修复问题。
> - **生产模式** 类似于机器交付给客户后的状态——一切精简、优化，追求最佳性能。

---

## 开发模式（Development Mode）

Expo 在开发模式下提供了出色的开发工具，包括：

- **Chrome 远程 JavaScript 调试**：通过浏览器中的 Chrome DevTools 远程调试你的 JavaScript 代码。
- **实时刷新（Live Reload）和热重载（Hot Reload）**：代码修改后自动刷新界面，无需手动重新加载。
- **元素检查器（Element Inspector）**：类似于浏览器的 DOM 检查器，可以检查 UI 元素的样式、布局等信息。

有关这些调试工具的详细使用方法，请参阅 Expo 的调试文档。

### 运行时检查与警告

开发模式还会执行一系列运行时检查（Runtime Checks），以显示警告信息，例如：

- 通知你使用了**已弃用的属性（Deprecated Property）**。
- 提醒你组件缺少**必需的 Props**。

这些检查能帮助你在开发阶段尽早发现潜在问题，避免它们流入生产环境。

> **初学者提示**：
> - **Props（属性）**：React 组件的输入参数，类似于 HTML 标签的属性。例如 `<Image source={...} />` 中的 `source` 就是一个 prop。
> - **已弃用（Deprecated）**：指某个功能虽然目前仍可使用，但已被官方标记为"不推荐使用"，未来版本可能会被移除。

元素检查器和性能监视器（Performance Monitor）在 Android 模拟器和 iOS 模拟器上均可使用。

### 性能影响

> **注意**：开发模式是有代价的——**你的应用在开发模式下运行速度较慢**。

这是因为开发模式额外执行了运行时检查、警告生成、实时刷新等功能，这些都会消耗额外的计算资源。

因此，**当你需要评估应用的实际性能时，请务必关闭开发模式**。

切换模式后，你需要**关闭并重新打开应用**，更改才会生效。

> **基于经验建议**：不要仅依赖开发模式下的体感速度来判断应用性能。很多开发者在开发模式下觉得"应用有点卡"，就急着做性能优化，结果切换到生产模式后发现性能完全没问题。建议始终在生产模式下进行性能基准测试。

### 访问开发者菜单

开发者菜单（Developer Menu）提供了众多简化开发和故障排查的功能。有关如何在 Android 和 iOS 平台上打开开发者菜单的说明，请参阅开发者菜单文档。

## 生产模式（Production Mode）

生产模式主要在以下两种场景中特别有用：

1. **评估真实性能**：由于开发模式会显著降低运行速度，切换到生产模式才能看到接近最终用户的真实表现。
2. **排查仅在生产环境中出现的缺陷**：某些 Bug 只在生产模式下才会显现（例如与代码压缩相关的问题），使用生产模式本地运行可以高效定位这些缺陷。

### 启动生产模式

要准确模拟最终用户的执行环境，请根据你使用的包管理器在终端中执行以下命令：

```sh
# npm
npx expo start --no-dev --minify

# yarn
yarn expo start --no-dev --minify

# pnpm
pnpm expo start --no-dev --minify

# bun
bun expo start --no-dev --minify
```

### 命令参数说明

| 参数 | 说明 |
|------|------|
| `--no-dev` | 关闭开发模式。指示 Metro 打包器（Bundler）将 `__DEV__` 环境变量设置为 `false`。 |
| `--minify` | 启用代码压缩。移除不必要的数据，如注释（Comments）、多余的空格和格式（Formatting）、以及未使用的代码（Unused Code）。 |

> **初学者提示**：
> - **`__DEV__`**：这是 Expo/React Native 提供的一个全局环境变量。在开发模式下它的值为 `true`，在生产模式下为 `false`。你可以在代码中使用它来区分不同环境下的行为，例如仅在开发模式下输出调试日志。
> - **Metro 打包器（Bundler）**：Expo 和 React Native 使用的 JavaScript 打包工具。它负责将你的所有源代码打包成可在设备上运行的文件。
> - **代码压缩（Minification）**：将代码中的变量名缩短、移除空格和注释等，使文件体积更小，从而加快加载速度。

### `__DEV__` 变量使用示例

虽然官方文档未直接给出 JavaScript 代码示例，但以下是 `__DEV__` 变量的常见用法：

```javascript
if (__DEV__) {
  // 仅在开发模式下执行的代码
  console.log('当前处于开发模式，这是调试信息...');
}

// 根据环境选择不同的 API 地址
const API_URL = __DEV__
  ? 'http://localhost:3000/api'    // 开发环境使用本地 API
  : 'https://api.myapp.com';       // 生产环境使用线上 API
```

> **基于经验建议**：善用 `__DEV__` 变量来保护调试代码。例如，你可以用它来包裹详细的日志输出，这样在生产环境中这些日志就不会被执行，既保证了安全性，又避免了不必要的性能开销。但请注意不要过度依赖它来做敏感信息保护——代码压缩并不会加密你的代码。

### 生产模式排查崩溃问题

如果你的独立构建（Standalone Build）在生产环境中出现崩溃，使用生产模式本地运行可以加速根因分析（Root Cause Analysis）。你可以在本地复现问题，同时享受接近真实用户环境的执行条件。

有关完整的编译说明，请参阅 Android 和 iOS 应用的专门构建指南。

## 开发模式 vs 生产模式 对比总结

| 特性 | 开发模式 | 生产模式 |
|------|----------|----------|
| 启动方式 | `npx expo start`（默认） | `npx expo start --no-dev --minify` |
| `__DEV__` 值 | `true` | `false` |
| 代码压缩 | 否 | 是（需 `--minify` 参数） |
| 运行速度 | 较慢 | 接近真实用户环境 |
| 远程调试 | 支持 | 不支持 |
| 实时/热重载 | 支持 | 不支持 |
| 元素检查器 | 支持 | 不支持 |
| 运行时警告 | 显示 | 不显示 |
| 适用场景 | 日常开发、调试 | 性能评估、排查生产环境问题 |

> **基于文档内容推导**：开发模式和生产模式的核心差异在于 `__DEV__` 全局变量的值和 Metro 打包器的处理方式。开发模式为 `true` 时启用所有开发辅助功能；生产模式为 `false` 时关闭这些功能并进行代码优化。两种模式本质上是同一份代码在不同配置下的运行表现，而非两套不同的代码库。

## 常见问题与注意事项

### 切换模式后不生效？

切换开发/生产模式后，必须**完全关闭并重新打开应用**，更改才会生效。仅仅刷新页面是不够的。

### 开发模式下应用很卡是正常的

开发模式下的额外检查和功能会消耗资源，导致应用运行速度明显慢于生产模式。这是预期行为，不必过度担心。

> **基于经验建议**：在开发过程中，建议大部分时间使用开发模式以便快速迭代和调试。仅在以下场景切换到生产模式：
> - 进行性能测试和优化时
> - 排查仅在发布版本中出现的 Bug 时
> - 发布前的最终验证阶段

### 平台兼容性

文档确认调试工具（包括元素检查器和性能监视器）在 **Android 模拟器（Android Emulator）** 和 **iOS 模拟器（iOS Simulator）** 上均可正常使用。两种平台在开发/生产模式的行为上没有本质差异，主要的平台差异体现在编译和构建流程上（详见各自的构建指南）。
