# 推送通知常见问题与故障排除 (FAQ)

> **原文地址**：[https://docs.expo.dev/push-notifications/faq/](https://docs.expo.dev/push-notifications/faq/)
>
> **更新日期**：2025年12月5日

本文档汇总了在使用 `expo-notifications` 库和 Expo 推送通知服务时，开发者经常遇到的问题和常见陷阱，并提供了对应的解答与解决方案。

---

## Expo 推送通知服务常见问题

### 推送通知服务是否收费？

**Expo 的推送通知服务是完全免费的。** 你可以自由使用 Expo 的推送通知基础设施向你的用户发送通知，无需支付任何费用。

> **关键术语解释**：
> - **推送通知服务（Push Notification Service）**：一个中介服务器，负责接收你的应用后端发出的通知请求，并将其转发给 Apple（APNs）或 Google（FCM）的推送平台。

---

### 发送通知的数量限制

每个项目（Project）的发送上限为 **每秒 600 条通知**。如果超过这个限制，后续的发送请求将会失败，直到发送量降低。

文档强烈建议在服务端实现**限流（Throttling）**和**重试（Retry）机制**：

> **关键术语解释**：
> - **限流（Throttling）**：控制请求发送速率的机制，防止短时间内发出过多请求。例如，限制每秒最多发送 500 条通知，超出的请求进入等待队列。
> - **重试（Retry）**：当请求失败后，自动重新尝试发送的机制。通常配合"指数退避"策略使用（即每次重试间隔时间逐渐加长）。
> - **Node SDK**：Expo 官方提供的 Node.js 软件开发工具包，它已经内置了限流和重试功能，如果你使用该 SDK，无需自行实现。

> **基于经验建议**：如果你没有使用 Expo 的 Node SDK，建议在你的服务端代码中使用消息队列（如 Redis Queue、Bull）来控制通知发送速率，并在失败时加入指数退避重试逻辑。

---

### 是否必须使用 Expo 推送通知服务？

**不是。** Expo 的推送通知服务是可选的。你可以通过 `getDevicePushTokenAsync` 方法获取设备原生推送令牌（Native Device Push Token），然后使用第三方推送服务商，或者直接对接 Apple 的 APNs 和 Google 的 FCM。

> **关键术语解释**：
> - **`getDevicePushTokenAsync`**：`expo-notifications` 库提供的一个异步方法，用于获取设备在 APNs（iOS）或 FCM（Android）上的原生推送令牌。
> - **APNs（Apple Push Notification service）**：Apple 官方的推送通知服务。
> - **FCM（Firebase Cloud Messaging）**：Google 官方的推送通知服务（原名 GCM）。
> - **原生推送令牌（Native Push Token）**：由 APNs 或 FCM 分配给每台设备的唯一标识符，用于直接向该设备发送通知。

---

### 与通知服务的连接是否加密？

**是的。** 与 Apple 和 Google 推送服务之间的所有通信均使用 **HTTPS 加密**。

---

### 通知内容是否会被存储？

**通知的内容（Payload）不会被持久化存储到数据库中。** 通知内容仅在内存和消息队列中短暂存在，仅用于传输过程中，发送完成后即被丢弃。

> **关键术语解释**：
> - **Payload（有效载荷）**：通知中携带的实际数据，包括标题（title）、正文（body）、附加数据（data）等字段。
> - **消息队列（Message Queue）**：一种异步通信机制，通知在被发送前会短暂地存放在队列中等待处理。

---

### Expo 工作人员是否能看到通知内容？

Expo 工程师**仅在主动调试期间**（例如在断点处）可能查看到通知的有效载荷数据。在其他情况下，Expo 团队**无法访问**你的通知数据。

---

### 送达保证

Expo 推送通知服务提供的是**"至少一次"（At-Least-Once）** 送达保证，即尽力将通知送达到底层推送提供商（APNs / FCM）。

需要注意以下几点：

- 在极少数情况下，通知**可能被重复发送**或**丢失**
- 通知成功移交给 APNs/FCM 后，会生成一个**回执（Receipt）**来确认提供商已接收
- 从提供商到最终设备的送达，取决于 **Apple 和 Google 各自的送达策略**，Expo 无法控制

> **关键术语解释**：
> - **至少一次（At-Least-Once）**：一种消息传递保证，确保消息至少被传递一次，但可能会产生重复。与之对应的是"至多一次"（At-Most-Once）和"恰好一次"（Exactly-Once）。
> - **回执（Receipt）**：APNs/FCM 确认已接收通知后返回的确认凭证，可用于验证通知是否成功提交给推送提供商。
> - **Ticket（票据）**：Expo 在接收到你的发送请求后立即返回的确认标识，表示请求已被接受并进入处理队列。Ticket 和 Receipt 是两个不同的概念：Ticket 表示"Expo 已收到"，Receipt 表示"APNs/FCM 已收到"。

> **基于文档内容推导**：由于存在"至少一次"的重复可能，你的客户端应用应具备通知去重能力，例如通过通知中的唯一标识符来判断是否已经处理过该通知。

---

### `ExpoPushToken` 何时以及为何会发生变化？

| 场景 | Token 是否变化 |
|------|--------------|
| 应用更新（App Update） | **不变** — Token 在应用更新时保持不变 |
| Android 卸载重装 | **可能变化** — 重新安装后 Token 可能会改变 |
| iOS 卸载重装 | **不变** — iOS 的 Token 在卸载重装后依然保留 |
| 修改 `applicationId`（Android） | **变化** — 会生成新的 Token |
| 修改 `experienceId` | **变化** — 会生成新的 Token |
| Token 自然过期 | **不会** — Token 不会自然过期 |
| 用户卸载应用 | 触发 **`DeviceNotRegistered`** 错误 — 你应当停止向该 Token 发送通知 |

> **关键术语解释**：
> - **`ExpoPushToken`**：Expo 推送通知服务分配给每台设备的令牌，格式类似 `ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]`，用于通过 Expo 服务向特定设备发送通知。
> - **`applicationId`**：Android 应用的包名标识符（如 `com.example.myapp`），在 `app.json` 中配置。
> - **`experienceId`**：Expo 项目的唯一标识符，用于区分不同的 Expo 应用。
> - **`DeviceNotRegistered`**：当目标设备已卸载应用或 Token 无效时，推送服务返回的错误代码。收到此错误后，应从你的数据库中移除该 Token。

> **基于文档内容推导**：建议在每次应用启动时调用获取 Token 的方法，并将最新的 Token 同步到你的服务器，以确保 Token 数据库始终是最新的。

## 推送通知故障排除

### 通知不工作

通知不工作的原因有很多，因为整个推送链路涉及多个组件（客户端、服务器、Expo 服务、APNs/FCM）。排查步骤如下：

1. **检查推送票据（Push Ticket）和回执（Receipt）**：查看其中是否包含错误信息
2. **测试本地通知（Local Notification）**：通过发送本地通知来验证客户端的通知处理逻辑是否正常。如果本地通知可以正常显示，说明客户端代码没有问题，问题出在服务端或凭证配置上

> **基于经验建议**：本地通知测试是排查问题的第一步。它能帮你快速区分问题是在客户端还是服务端，大幅缩小排查范围。

#### 使用终端命令快速获取推送回执

**第一步：发送一条测试通知**

```sh
curl -H "Content-Type: application/json" -X POST "https://exp.host/--/api/v2/push/send" -d '{
  "to": "ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]",
  "title":"hello",
  "body": "world"
}'
```

> **关键术语解释**：
> - 将 `ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]` 替换为你实际设备的 Expo 推送令牌。
> - 该请求会返回一个包含 `id` 字段的 JSON 响应，这个 `id` 就是**推送票据 ID（Push Ticket ID）**。

**第二步：使用票据 ID 获取推送回执**

```sh
curl -H "Content-Type: application/json" -X POST "https://exp.host/--/api/v2/push/getReceipts" -d '{
  "ids": [
    "XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX"
  ]
}'
```

> **关键术语解释**：
> - 将 `XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX` 替换为第一步返回的实际票据 ID。
> - 回执（Receipt）中会包含通知的送达状态和可能的错误详情。如果回执中包含 `errors` 字段，说明通知在送达 APNs/FCM 时出了问题。

---

### 开发模式下通知正常，但发布模式下不工作

如果通知在开发环境（Development）下工作正常，但在发布/生产环境（Production / Release）下失效，通常是因为**生产环境的推送凭证缺失或配置错误**。

需要注意以下关键信息：

- **SDK 53 及更高版本**：`Expo Go` 客户端不支持推送通知，必须使用**开发构建（Development Build）** 来测试。这意味着你需要通过 EAS Build 创建自定义的构建版本。
- **旧版 SDK（53 之前）**：可以使用 Expo 的默认凭证。
- **应用商店构建（Store Builds）**：需要你配置自己的推送凭证：
  - **Android**：按照 FCM 凭证配置指南设置 Firebase Cloud Messaging
  - **iOS**：通过 `eas credentials` 命令管理推送密钥（Push Key）。**注意：如果撤销（Revoke）了 iOS 推送密钥，通知将无法送达，直到你创建新的密钥**

> **关键术语解释**：
> - **开发构建（Development Build）**：通过 EAS Build 生成的自定义应用构建版本，包含了原生代码，支持推送通知等原生功能。与 Expo Go 不同，它是一个完整的独立应用。
> - **Expo Go**：Expo 提供的通用客户端应用，由于安全限制，从 SDK 53 起不再支持推送通知功能。
> - **推送密钥（Push Key）**：iOS 平台上用于授权你的服务器向 APNs 发送通知的密钥文件（`.p8` 格式）。
> - **`eas credentials`**：EAS CLI 提供的凭证管理命令，用于查看和管理项目的推送密钥、证书等。

> **基于经验建议**：在提交到应用商店之前，务必在发布模式的开发构建上测试推送通知。很多开发者在开发阶段一切正常，上线后才发现凭证问题。

---

### Android 上通知偶尔无法送达

Android 上通知间歇性无法送达，通常与**消息优先级（Priority）** 设置有关。

Expo 推送通知 API 支持以下优先级值：

| 优先级值 | 说明 |
|---------|------|
| `"default"` | 默认优先级 |
| `"high"` | 高优先级 — 最有可能触发设备显示通知 |
| `"normal"` | 普通优先级 |
| 省略不填 | 等同于 `"default"` |

> **基于经验建议**：如果你希望通知在 Android 上可靠地显示（尤其是时间敏感的通知），建议将优先级设为 `"high"`。Android 系统会根据优先级决定通知的展示方式（例如是否发出声音、是否弹出横幅等）。

---

### 处理过期的推送通知凭证

当 iOS 推送密钥过期时，需要重新生成。操作步骤如下：

1. 运行 `eas credentials` 命令
2. 选择 **iOS** 平台
3. 选择对应的构建配置文件（Profile）
4. **删除**旧的推送密钥
5. **创建**新的推送密钥

```sh
eas credentials
```

---

### iOS 报 "No valid aps-environment entitlement string found" 错误

该错误表示项目**缺少推送密钥（Push Key）**。

**解决步骤**：

1. 在 [项目凭证面板（Project Credentials Dashboard）](https://expo.dev) 上验证是否已配置推送密钥
2. 通过 EAS CLI 发起一次 iOS 构建来自动生成新的推送密钥：

```sh
eas build --profile [profile] --platform ios
```

> **关键术语解释**：
> - **`aps-environment`**：iOS 的一个权限字符串（Entitlement），用于声明应用是否有权使用推送通知服务。该字符串的值由苹果在签名时自动注入，取决于你使用的证书类型（`development` 或 `production`）。
> - **`[profile]`**：EAS 构建配置文件的名称（如 `development`、`preview`、`production`），定义在 `eas.json` 文件中。

> **基于经验建议**：如果你在 Apple Developer 后台手动管理证书和密钥，请确保推送密钥与应用使用的 Provisioning Profile 匹配，否则会持续出现此错误。

---

### 发送通知时出现错误消息

检查推送票据（Ticket）或回执（Receipt）中的 `details` 字段，其中包含具体的错误代码和描述信息。

> **关键术语解释**：
> - **`details` 字段**：回执或票据中与错误相关的附加信息字段。它通常包含一个 `error` 属性，例如 `"DeviceNotRegistered"`、`"InvalidCredentials"`、`"MessageTooBig"` 等，帮助你定位具体问题。

---

### iOS 上获取推送令牌耗时过长

在 iOS 上，调用 `getDevicePushTokenAsync` 或 `getExpoPushTokenAsync` 可能会花费较长时间才能返回结果。这通常不是代码错误，而是由以下因素导致的：

- **网络连接不稳定**
- **设备处于飞行模式**
- **APNs 服务器暂时不可用**

Apple 官方建议将此视为一种**非错误情况**，在应用中优雅地处理 — 如果获取令牌超时，应**禁用依赖推送的功能**，而非崩溃或阻塞。

以下是社区总结的排查和解决方法：

#### 1. 查阅 Apple 官方技术说明

确保设备有稳定的互联网连接。如果网络防火墙阻止了 **端口 5223**，需要解除阻止（该端口用于 APNs 的长连接通信）。

对于裸 React Native 项目（Bare React Native），需要**手动在 Xcode 中启用推送通知能力（Push Notification Capability）**。

#### 2. 稍后重试

APNs 服务器可能暂时不可用。等待一段时间后重试，有时甚至需要等待几天（Apple 服务器维护时）。

#### 3. 关闭设备上的网络共享

关闭 iPhone 的"个人热点"或网络共享功能。

#### 4. 重启设备

重启设备，特别是在切换构建类型之后（例如从 Xcode 直连切换到 TestFlight 分发）。

#### 5. 插入 SIM 卡

在未配置 SIM 卡的设备上，有时会出现无法获取推送令牌的 Bug。插入一张有效的 SIM 卡可以解决此问题。

> **基于经验建议**：在应用代码中为 `getDevicePushTokenAsync` 添加超时处理和用户友好的提示，例如"推送通知暂时不可用，请稍后再试"，避免因令牌获取失败导致应用启动卡死。

## 其他

### 直接通过 FCM 和 APNs 发送通知

如果你不想使用 Expo 的推送通知服务，希望直接与 FCM 和 APNs 通信，请参考 Expo 文档中关于"通过 FCM 和 APNs 直接发送通知"的指南。

> **基于文档内容推导**：直接对接 FCM 和 APNs 意味着你需要自行处理推送令牌的收集、消息格式构建、认证和错误重试等逻辑，适合对推送通知有特殊需求（如自定义加密、超低延迟要求）的项目。

---

### Android 通知图标显示为灰色或白色方块

如果通知图标显示为一个实心的灰色或白色方块，说明你使用的**图片资源不符合要求**。

Google 要求 Android 通知图标必须是一个**纯白色图片，背景为透明**。

> **关键术语解释**：
> - **通知图标（Notification Icon）**：在 Android 状态栏和通知抽屉中显示的小图标。Android 系统会自动将通知图标处理为单色显示，因此如果使用彩色图片，会被渲染为一个纯色方块。

> **基于经验建议**：使用 Android Studio 的 Image Asset Studio 工具来生成符合规范的通知图标。图标应为 24dp x 24dp 的纯白色 PNG 图片，所有需要显示的部分为白色，需要隐藏的部分为透明。不要使用带有颜色或灰度的图片。
