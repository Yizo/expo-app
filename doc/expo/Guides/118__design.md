# Expo Modules API：设计考量

本文档阐述了 Expo Modules API 背后的设计理念与架构决策，解释了为什么 Expo 团队选择以当前这种方式来构建原生模块系统。

---

## 背景介绍

管理大量的原生库使得维护原生模块在不断变化的生态系统中变得十分困难。Expo 团队开发了一套强大的工具集，旨在使这些库的构建和维护变得更加轻松。

> **核心问题**：Expo 团队需要同时管理超过 50 个原生 SDK 组件，每个组件都需要适配 iOS 和 Android 两个平台。在如此大规模的维护工作中，他们发现了大量重复出现的问题模式，这些问题推动他们重新思考模块系统的设计。

---

## 充分利用现代语言特性

经过多年管理超过 50 个原生 SDK 组件的经验，团队发现**许多问题都是由未处理的空值（null）或错误的类型（type）引起的**。

### 为什么选择 Swift 和 Kotlin？

> **关键术语解释**：
> - **空值（Null）**：表示变量没有值的状态。在旧式语言中，未处理的空值常导致程序崩溃。
> - **可选类型（Optional Types）**：Swift 等现代语言提供的类型系统特性，强制开发者显式处理值可能为空的情况，从而在编译阶段就能捕获潜在的空指针错误。
> - **Objective-C**：Apple 的传统 iOS 开发语言，具有高度动态性，但缺乏现代类型安全特性。
> - **Swift**：Apple 推出的现代 iOS 开发语言，拥有强大的编译期类型检查。

Objective-C 的动态特性以及缺失的可选类型（Optional Types）使得某些 bug 很难被检测到，而 Swift 的编译器可以轻松捕获这些问题。

> **基于经验建议**：如果你正在开发新的 Expo 模块，强烈建议直接使用 Swift（iOS）和 Kotlin（Android）而非 Objective-C 和 Java。这不仅是 Expo 团队的设计方向，也能显著减少因类型不安全导致的运行时错误。

### 降低上下文切换的成本

在不同原生语言之间频繁切换上下文（context switching）是另一个挑战。虽然由于平台差异，完全避免上下文切换是不可能的，但创建者希望提供**统一的 API 和文档**来简化开发流程，帮助单个开发者高效地维护跨平台库。

因此，整个生态系统从底层开始就被设计为使用现代原生语言来构建：**Swift（iOS 端）和 Kotlin（Android 端）**。

## 简化运行时之间的数据传递

### 自动参数验证与转换

> **关键术语解释**：
> - **运行时（Runtime）**：程序实际执行时的环境。JavaScript 有自己的运行时，iOS 原生代码有自己的运行时，它们之间传递数据需要特殊的桥接机制。
> - **NSDictionary**：iOS/Objective-C 中的字典（键值对集合）类型，类似于 JavaScript 的对象。
> - **ReadableMap**：React Native Android 端用于表示 JavaScript 对象的数据结构。
> - **Record**：Expo Modules API 提供的原生结构体概念，允许将 JavaScript 字典映射为类型安全的原生结构。

系统完全了解预期的原生参数类型，因此可以**自动预验证并转换参数**。此外，字典可以被转换为被称为 **Record** 的原生结构体。

团队解决的一个重大问题是 JavaScript 传递到原生代码时的参数验证。在使用 `NSDictionary` 或 `ReadableMap` 等泛型映射时，这个过程**特别容易出错、耗时且难以维护**，因为在运行时值的类型是未知的，需要手动验证每个属性。

### 类型感知带来的自动转换

通过了解预期类型，系统可以**自动将参数转换为平台特定的格式**。例如：

> **基于文档内容推导**：JavaScript 中的坐标对象 `{x: 10, y: 20}` 可以被自动转换为 iOS 平台的 `CoreGraphics` 的 `CGPoint`，或者 Android 平台对应的 `Point` 类型，开发者无需手动编写转换代码。

### 支持的类型体系

总体而言，该框架提供了**强大且可扩展的类型转换和类型安全**机制。原生支持的类型包括：

- **基本类型（Primitives）**：字符串、数字、布尔值等基础数据类型
- **复杂内置类型（Complex Built-in Types）**：如数组、字典、URL 等系统预定义类型
- **用户自定义 Record（User-defined Records）**：开发者自行定义的结构体映射
- **枚举（Enums）**：可映射 JavaScript 常量到原生枚举值

> **基于经验建议**：在设计模块的 API 接口时，优先使用 Record 而非原始字典来传递结构化数据。Record 提供编译期类型检查，能在开发阶段就发现参数错误，而字典则只能在运行时发现问题。

## 支持表达力强的面向对象 API

### Shared Objects（共享对象）

> **关键术语解释**：
> - **Shared Objects（共享对象）**：Expo Modules API 提供的一种机制，允许开发者在原生端维护对象状态，同时让 JavaScript 端可以引用和操作这些对象，而无需在两端手动同步状态。
> - **单一状态源（Single Source of Truth）**：软件设计原则，指数据只在一个地方维护，避免多处同步导致的数据不一致问题。

开发者应该将原生模块的状态维护在**单一位置**，而不是将状态分散在 JavaScript 和原生代码之间并手动管理同步。这种能力被称为 **Shared Objects**。

一个具体的例子是：`expo-sqlite` 的数据库实例就是基于 Shared Objects 实现的。

> **基于经验建议**：如果你的模块涉及需要维护复杂内部状态的原生资源（如数据库连接、媒体播放器、硬件传感器等），应优先考虑使用 Shared Objects 模式，而不是尝试在 JavaScript 层复制和同步状态。

## 提供安全且可组合的生命周期事件钩子

> **关键术语解释**：
> - **生命周期（Lifecycle）**：应用从启动到关闭过程中经历的各个阶段（如创建、恢复、暂停、销毁等）。开发者常需要在这些阶段执行特定逻辑。
> - **AppDelegate**：iOS 应用中负责管理应用生命周期的核心类。
> - **MainActivity**：Android 应用的主 Activity 类，是应用与系统交互的主要入口。
> - **Continuous Native Generation（持续原生生成）**：Expo 的一种构建模式，每次构建时自动重新生成原生项目代码，使得原生配置始终保持最新。

### 解耦的生命周期管理

- **Android 端**：生命周期监听器（Lifecycle Listeners）
- **iOS 端**：AppDelegate 订阅者（AppDelegate Subscribers）

这些功能让开发者可以**钩入应用的生命周期**，而无需将模块代码分散到 `MainActivity` 和 `AppDelegate` 类中。

这种方式对于 **Continuous Native Generation** 特别有益，它提供了一种**可组合的方式**来独立管理生命周期事件，而不受其他库的干扰。

> **基于文档内容推导**：传统做法中，如果一个应用安装了多个需要监听生命周期的原生模块，每个模块都需要修改 MainActivity 或 AppDelegate，这在使用 Continuous Native Generation 时会导致严重冲突。Expo 的生命周期监听器机制通过提供统一的注册和分发系统，彻底解决了这个问题。

## 支持新架构同时保持向后兼容

### 新架构带来的挑战

> **关键术语解释**：
> - **New Architecture（新架构）**：React Native 从 0.68 版本引入的全新底层架构，旨在提升性能和开发体验。
> - **Turbo Modules**：新架构的一部分，提供延迟加载原生模块的能力，提升应用启动速度。
> - **Fabric**：新架构中的新渲染系统，替代了旧的 UI Manager。
> - **C++**：一种底层编程语言，React Native 新架构的核心使用 C++ 实现，以实现跨 iOS 和 Android 的代码共享。
> - **Renderer-agnostic（渲染器无关）**：不依赖于特定渲染系统的设计理念，使模块可以在不同的渲染架构下正常工作。

React Native 0.68 引入了新架构，包括 **Turbo Modules** 和 **Fabric** 渲染系统。适配 Fabric 需要额外的工作量，因为它缺少兼容层——旧的 View Manager 无法在 Fabric 下工作，反之亦然。这迫使库同时支持两套系统，**增加了技术债务**。

### C++ 的现实问题

由于新架构大量依赖 C++，开发者可能**需要编写 C++ 代码**。然而，React Native 开发者更倾向于使用高层级的 JavaScript，对于编写 C++ **相当不情愿**。此外，引入 C++ 还会带来以下负面影响：

- **构建速度下降**：特别是在 Android 平台上，C++ 的编译会显著增加构建时间
- **调试复杂度增加**：C++ 的调试工具和调试体验远不如 JavaScript 友好

### Expo 的解决方案：渲染器无关设计

为了应对上述挑战，Expo 框架被设计为**渲染器无关（renderer-agnostic）**的。这意味着：

- 模块**无需了解**应用使用的是新架构还是旧架构
- 显著**降低了库开发者的负担**
- 开发者可以专注于业务逻辑，而不必操心底层渲染架构的差异

> **基于经验建议**：如果你正在开发 Expo 模块，无需担心新架构兼容性问题。Expo 的渲染器无关设计意味着你的模块代码可以同时在旧架构和新架构下工作，大大减少了适配成本。这也是选择 Expo Modules API 而非直接编写 React Native 原生模块的重要优势之一。

---

## 设计理念总结

| 设计目标 | 解决的问题 | 实现方式 |
|---------|-----------|---------|
| 利用现代语言特性 | 空值/类型错误、上下文切换困难 | 从底层支持 Swift 和 Kotlin |
| 简化跨运行时数据传递 | 参数验证易出错、类型转换繁琐 | 自动预验证、自动类型转换、Record |
| 支持面向对象 API | 状态分散、同步困难 | Shared Objects 共享对象机制 |
| 可组合的生命周期钩子 | 生命周期代码分散、库间冲突 | 生命周期监听器和 AppDelegate 订阅者 |
| 新架构兼容 | Fabric 适配成本高、C++ 开发门槛高 | 渲染器无关（renderer-agnostic）设计 |
