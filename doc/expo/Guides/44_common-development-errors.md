# Expo 常见开发错误排查

## 文档解决的问题

这篇文档解决的是：Expo 开发过程中一些高频错误各自通常意味着什么，以及第一轮应该怎么排查。

## 适用场景

- 项目启动失败、连接失败、版本不匹配、注册失败等。
- 你需要一份“错误信息 -> 常见原因 -> 初步排查动作”的对照表。
- 你想快速判断问题更可能出在网络、缓存、版本、原生注册还是构建链路。

## 核心概念

这页本质上不是教程，而是故障字典。每个错误项都由两部分组成：

- 为什么会发生
- 优先尝试什么排查方式

## 按原文结构整理的核心内容

### 1. `Metro bundler ECONNREFUSED 127.0.0.1:19001`

含义：

- 本地开发服务器连接失败。

文档建议：

- 删除 `.expo` 本地状态目录
- 检查防火墙
- 检查代理配置

### 2. `Module AppRegistry is not a registered callable module (calling runApplication)`

含义：

- JS bundle 在启动时没有成功执行，通常前面已经有别的代码错误挡住了。

文档建议：

- 运行 `npx expo start --no-dev --minify` 复现生产 bundle
- 连接设备并查看 Android Studio / Xcode 日志
- 检查 Babel 配置
- 少数情况下也可能与 Metro 压缩器兼容性有关

### 3. `npm ERR! No git binary found in $PATH`

含义：

- 没装 git，或 git 没进系统 `PATH`

文档建议：

- 安装 git
- 检查系统 PATH 配置

### 4. `XX.X.X is not a valid SDK version`

含义：

- 当前 SDK 已弃用或不再支持。

文档建议：

- 升级项目到受支持的 SDK
- 如果项目本身版本没问题，检查 Expo Go 是否过旧

### 5. `React Native version mismatch`

含义：

- 终端里的开发服务器正在打包一个 React Native 版本，但设备上的 App 壳是另一个版本。

文档建议：

- 检查 `app.json` 和 `package.json` 中的版本是否对齐

### 6. `Application has not been registered`

含义：

- 原生侧和 JS 侧注册的 AppKey 不一致。

文档建议：

- 对齐 AppKey

### 7. `Application not behaving as expected`

含义：

- 可能是缓存导致你看到的不是当前状态。

文档建议：

- 清理各类缓存

## 关键命令、配置、文件说明

关键命令：

- `rm -rf .expo`
- `npx expo start --no-dev --minify`

关键文件或配置：

- `.expo`：本地状态目录
- `app.json`
- `package.json`
- Babel 配置文件

## 注意事项、限制条件和坑点

- 一条错误信息背后未必只有一种根因，这页给的是“高频原因和首轮排查建议”。
- `AppRegistry` 报错往往是“前面还有更早的错误”，不要只盯着表面报错。
- 版本不匹配不一定是依赖装错，也可能是设备里的 Expo Go 版本落后。
- 缓存问题很常见，但“先清缓存”不等于可以跳过版本、配置和日志检查。

## React Web 开发者易误解点

- 容易把所有启动错误都当成 JS 语法问题。这里很多问题来自网络、原生壳版本或 App 注册入口。
- 容易把 Expo Go 视为“永远最新的运行容器”。当前文档明确提示它本身也可能版本过旧。
- 容易忽略原生日志价值。`AppRegistry` 这类问题，文档建议直接去 Android Studio / Xcode 看更完整堆栈。

## 实际开发建议

- 先按错误文案分类，不要一上来就做大范围重装。
- 对启动类错误，优先同时检查：开发服务器、原生日志、版本对齐、缓存。
- 基于经验建议：团队可以把这页里的常见错误整理成内部排障 checklist，减少重复沟通。

- 每个错误都给出成因解释和调试建议。
- 某些问题需要查看原生日志而不是只看终端。

## 基于文档内容推导

- 基于文档内容推导：Expo 开发问题很多是“运行环境不一致”导致的，而不只是业务代码错误。
- 基于文档内容推导：把错误分成网络、版本、注册、缓存、构建几个类别，会比按表象乱试更高效。
