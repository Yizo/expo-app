# Expo 的开发模式与生产模式

## 文档解决的问题

这篇文档解决的是：Expo 项目在 development mode 和 production mode 下有什么区别，什么时候应该切换，以及如何本地模拟生产模式。

## 适用场景

- 你想理解为什么开发时 App 比正式环境慢。
- 你要排查“只在生产环境出现”的 bug。
- 你要做性能测试，不想被开发模式误导。

## 核心概念

- `development mode`：本地开发默认模式，包含警告和调试工具。
- `production mode`：更接近真实用户环境，代码会被压缩，`__DEV__` 为 `false`。
- `--minify`：压缩 bundle，移除注释、格式化和未使用代码等。

对 React Web 开发者来说，它类似于开发构建和生产构建的差异，但在移动端差异对性能、调试体验和运行行为的影响更明显。

## 按原文结构整理的核心内容

### 1. 默认运行模式

文档明确说：

- 本地执行 `npx expo start` 默认是开发模式
- 发布后的项目，例如 `eas update` 产物或 standalone app，会运行在生产模式

### 2. 开发模式提供什么

开发模式包含：

- 各种警告和校验
- 远程 JavaScript 调试
- live reload
- hot reloading
- 类似 Chrome inspector 的元素检查能力

这使开发体验更好，但文档也明确强调：它的代价是运行更慢。

### 3. 为什么性能测试不能在开发模式做

页面专门强调：

- 开发模式会明显拖慢 App
- 测性能时必须关闭开发模式
- 切换后需要关闭并重新打开 App 才会生效

这说明如果你拿开发环境帧率或启动速度做性能判断，结论很可能不可靠。

### 4. 如何本地模拟生产模式

最直接的命令是：

```sh
npx expo start --no-dev --minify
```

这里包含两个意思：

- `--no-dev`：让 JS 以生产模式运行
- `--minify`：让 bundle 被压缩

文档说这对排查只在 standalone app 中出现的错误很有帮助。

## 关键命令、配置、文件说明

关键命令：

- `npx expo start`
- `npx expo start --no-dev --minify`

关键运行时变量：

- `__DEV__`

当前文档未涉及具体配置文件。

## 注意事项、限制条件和坑点

- 开发模式性能慢是正常现象，不一定代表代码有性能问题。
- 切模式后需要重启 App 才生效。
- 只在开发模式验证通过，不代表生产模式一定没问题。
- 只在生产模式跑，不利于调试效率。

## React Web 开发者易误解点

- 容易把“本地跑起来”直接当成性能参考。移动端开发模式的额外开销往往比 Web 更明显。
- 容易忽视 `__DEV__` 这个运行时分支带来的行为差异。
- 容易以为 `--minify` 只是压缩体积。文档明确指出它也能帮助复现正式环境 bug。

## 实际开发建议

- 日常开发保持开发模式，性能测试和发布前验收切到生产模式模拟。
- 遇到“只在线上崩”或“只在线上白屏”时，第一时间尝试 `--no-dev --minify`。
- 基于经验建议：任何性能结论都至少在生产模式模拟下复核一次。

## 文档明确说明

- `npx expo start` 默认运行开发模式。
- 发布后的应用运行生产模式。
- 开发模式包含调试工具和额外校验，但更慢。
- 可用 `npx expo start --no-dev --minify` 本地模拟生产行为。

## 基于文档内容推导

- 基于文档内容推导：开发模式和生产模式不仅仅是“有无 sourcemap”差异，而是会直接影响问题是否能复现。
- 基于文档内容推导：团队如果不建立“生产模式复核”的习惯，容易在上线前漏掉真实环境问题。
