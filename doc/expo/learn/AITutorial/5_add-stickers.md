# 添加贴纸：实现 Emoji 选择、拖动与缩放

> 原文档更新时间：2026 年 6 月 10 日  
> 所属教程：Build with AI，第 4 章

## 文档解决的问题

本章指导你通过 AI 编程 Agent，为照片装饰应用增加以下能力：

1. 选中照片后，从“选择照片”模式切换到“装饰照片”模式。
2. 从底部弹出的面板中选择 Emoji。
3. 将 Emoji 作为贴纸叠加到照片上。
4. 用手指拖动贴纸。
5. 双击贴纸，在两种尺寸之间切换。
6. 让拖动和缩放动画保持流畅。

本章适合已经完成照片选择功能、准备增加图片装饰交互的场景。它只处理贴纸的选择与操作，不负责保存最终图片；保存功能属于下一章。

## 阅读前需要理解的背景

### AI Agent 在本章中的角色

本章不是要求你手动逐行实现功能，而是提供自然语言提示词，让 AI Agent 修改项目代码。

Agent 能够：

- 阅读现有代码；
- 根据需求创建或修改组件；
- 引入手势和动画库；
- 调整界面状态及交互逻辑。

但 Agent 无法亲自触摸设备，因此不能判断拖动是否跟手、双击是否容易触发或动画是否自然。这些体验必须由开发者在设备或模拟器上验证。

### Modal 与底部滑出面板

Modal 可以理解为覆盖在当前页面之上的独立界面层。本章要求它从屏幕底部滑出，用来展示可选 Emoji。

对 React Web 开发者而言，它在用途上接近：

- 弹窗；
- Drawer；
- Bottom Sheet；
- 通过 Portal 渲染的浮层。

需要注意：文档只描述了“从底部滑出的 modal”，没有明确指定使用哪个 Modal 或 Bottom Sheet 组件，也没有提供动画实现细节。

### 横向滚动列表

Emoji 需要排成横向列表，并允许用户左右滑动浏览。这类似 Web 中的横向滚动容器，但移动端主要通过手指滑动操作。

### 手势与动画库

文档明确指定使用：

- `react-native-gesture-handler`：识别拖动、双击等原生触摸手势；
- `react-native-reanimated`：实现平滑的位置和尺寸动画。

可以将前者理解为“手势输入层”，后者理解为“动画执行层”。二者共同完成贴纸交互，而不是依靠浏览器中的鼠标事件、DOM 事件或 CSS Transition。

## 实现流程

### 第一步：添加 Emoji 选择器

将下面的需求交给 AI Agent：

> 当用户已经选择照片时，无论照片来自系统相册，还是来自 “Use this photo” 操作，都将原来的两个按钮替换为一行新选项：
>
> - 左侧显示 “Reset”，用于恢复原来的按钮；
> - 中间显示一个圆形 “+” 按钮；
> - 点击 “+” 后，从底部滑出标题为 “Choose a sticker” 的 Modal；
> - Modal 中横向展示 `assets/images` 目录内的 Emoji 图片；
> - 点击某个 Emoji 后关闭 Modal，并把它作为贴纸显示在照片上方。

#### 界面状态变化

这一阶段实际包含两种界面模式：

| 模式 | 进入条件 | 界面内容 |
|---|---|---|
| 选择照片模式 | 尚未选择照片，或点击 Reset | 原来的两个按钮 |
| 装饰照片模式 | 已从相册选择照片，或点击 Use this photo | Reset 与圆形 + 按钮 |

虽然文档没有给出具体状态变量，但可以看出界面必须根据“是否已经选择照片”进行条件渲染。

#### Reset 的含义

文档明确说明 Reset 会恢复原来的按钮，但没有进一步说明它是否还应当：

- 清除当前照片；
- 清除已添加的贴纸；
- 重置贴纸位置或尺寸；
- 关闭已打开的 Modal。

因此，不应仅凭本章内容假定这些行为。实现后应检查 Agent 对 Reset 的解释是否符合整个应用已有逻辑。

#### 预期结果

完成后应看到：

1. 选择照片后，原来的按钮变为新操作栏。
2. 点击 `+`，Emoji 面板从底部滑入。
3. Emoji 列表可以横向滚动。
4. 点击 Emoji 后，面板关闭。
5. 选中的 Emoji 显示在照片之上。

这里的“显示在照片之上”意味着贴纸需要作为覆盖层参与布局，而不是插入到照片下方或普通文档流中。

### 第二步：让贴纸支持拖动与双击缩放

继续要求 AI Agent 使用 `react-native-gesture-handler` 和 `react-native-reanimated` 实现：

- 单指拖动贴纸，使其可以放置在照片的任意位置；
- 双击贴纸，将尺寸放大为原来的两倍；
- 再次双击，将贴纸缩回原始尺寸；
- 拖动和缩放过程应平滑执行。

#### 交互状态

贴纸至少存在两种尺寸状态：

| 当前状态 | 双击后的结果 |
|---|---|
| 原始尺寸 | 放大到两倍 |
| 两倍尺寸 | 恢复原始尺寸 |

这不是无限累积放大，而是在两种尺寸之间切换。

#### 预期结果

完成后需要手动验证：

- 贴纸能否跟随手指移动；
- 松手后是否停留在目标位置；
- 双击能否可靠触发；
- 第一次双击是否放大到两倍；
- 再次双击是否恢复；
- 动画是否顺畅。

## 文件、目录与依赖说明

### `assets/images`

文档要求从该目录加载 Emoji 图片。

这表示 Emoji 是项目内置的静态图片资源，不是：

- 系统 Emoji 键盘返回的字符；
- 在线下载的图片；
- 用户从相册选择的文件。

文档没有列出具体文件名、图片数量或资源加载方式。

### `react-native-gesture-handler`

用途：处理拖动和双击手势。

React Web 开发者不要将它简单等同于 `onMouseDown`、`onMouseMove` 或 `onDoubleClick`。移动端需要处理连续触摸坐标、手势竞争以及手势状态变化，专用手势库更适合这类交互。

### `react-native-reanimated`

用途：平滑更新贴纸的位置和尺寸。

它解决的是交互过程中的动画表现。文档没有说明具体使用哪些 API、如何组织共享动画状态，或如何配置这两个库。

- iOS 或 Android 原生工程配置；
- 权限配置；

## 关键注意事项与限制

### 必须覆盖两种照片来源

装饰模式需要在以下任一操作完成后出现：

- 从照片库选择照片；
- 点击 `Use this photo`。

如果实现只处理其中一条路径，就不符合文档要求。

### Emoji Modal 必须关闭

选择 Emoji 后需要同时完成两件事：

1. 关闭选择面板；
2. 将所选 Emoji 放到照片上。

只完成贴纸更新而保留 Modal，或只关闭 Modal 而未显示贴纸，都属于流程不完整。

### 列表方向是横向

Emoji 图片应在可横向滚动的列表中展示，而不是普通纵向列表。

### 双击是尺寸切换

文档要求在原始尺寸和两倍尺寸之间来回切换，不是每次双击继续放大。

### 流畅性必须人工检查

Agent 能检查代码结构，却不能感知真实触摸体验。重点观察以下异常：

- 拖动明显延迟；
- 贴纸突然跳到其他位置；
- 松手后回到原位；
- 位置发生吸附或回弹；
- 双击后尺寸突变、不流畅；
- 双击难以触发。

发生问题时，应向 Agent 描述具体可观察现象，而不是只说“手势有问题”。例如：

- “拖动开始时贴纸会向右下角跳一下。”
- “松手后贴纸会回到拖动前的位置。”
- “快速双击没有反应，慢速点击却会触发。”
- “放大后再次拖动，贴纸位置发生偏移。”

### 文档没有明确说明的限制

本章未说明：

- 是否支持同时添加多个贴纸；
- 是否限制贴纸只能在照片范围内移动；
- 放大后的贴纸能否超出照片边界；
- 是否支持旋转、捏合缩放或删除贴纸；
- 更换照片时是否保留贴纸；
- Reset 是否清除贴纸；
- Emoji 资源的尺寸和格式要求；
- 是否需要处理无障碍、横屏或不同屏幕尺寸；
- 手势库与动画库的具体版本要求。

这些行为不能视为本章已经定义。

## React Web 开发者最容易误解的地方

### 不存在 DOM 定位模型

“把 Emoji 放在照片上”不能直接套用 Web 中 `position: absolute`、DOM 层级和 CSS transform 的完整心智模型。React Native 使用自己的布局、样式与原生视图系统。

二者概念上相近，但具体组件、属性和坐标处理方式不同。

### 主要输入不是鼠标

拖动功能应围绕触摸手势设计。鼠标模拟可能帮助初步调试，但最终体验仍应通过真实手指操作确认。

### 动画不等同于 CSS Transition

React Native 不运行浏览器 CSS 动画。本章明确要求使用 Reanimated，说明贴纸的位置和缩放需要通过 React Native 动画机制实现。

### 双击不是浏览器 `dblclick`

React Native 手势需要识别连续点击的次数和时间关系。手势识别还可能与拖动产生竞争，例如轻微移动可能被识别为拖动而不是点击。这也是必须实际测试的原因之一。

### Modal 不是浏览器弹窗

它不是 `window.alert()`，也不等同于普通 DOM 对话框。它是移动应用界面中的覆盖层，本章还要求具备从底部滑入的视觉行为。

### React 状态正确不代表触摸体验正确

在 Web 项目中，开发者可能更关注状态更新和渲染结果；本章还要求验证连续手势中的响应速度、位置连续性和动画反馈。代码逻辑正确只是完成条件的一部分。

## 实际开发中的使用方式

### 推荐验收顺序

以下顺序是**基于文档内容推导**出的实用验收方式：

1. 分别通过相册和 `Use this photo` 进入装饰模式。
2. 检查原按钮是否被 Reset 和 `+` 替换。
3. 点击 `+`，检查 Modal 的标题和底部滑入效果。
4. 横向滑动 Emoji 列表。
5. 选择 Emoji，检查 Modal 是否关闭、贴纸是否覆盖在照片上。
6. 将贴纸向多个方向拖动，并检查松手后的最终位置。
7. 连续双击两次，验证放大与恢复。
8. 在放大状态下继续拖动，观察位置是否跳变。
9. 发现异常时，将具体现象和复现步骤反馈给 Agent。

### 与 Agent 沟通的重点

本章体现的开发方式是“提出目标、运行验证、描述偏差、继续修正”。

需求提示应尽量包含：

- 触发条件；
- 操作方式；
- 期望视觉结果；
- 状态切换规则；
- 出错时的具体现象。

仅说“增加贴纸功能”无法完整表达本章要求，因为它没有说明 Modal、资源目录、横向列表、拖动、双击和动画行为。

### 后续功能边界

本章完成后，应用中的贴纸只存在于当前编辑界面。文档明确指出，还缺少“保留创作结果”的能力；下一章才会把装饰后的照片保存到手机照片库。

因此，“贴纸显示正确”和“最终合成图片已经保存”是两个不同阶段，不能混为一谈。

## 总结

本章将应用从“选择照片”推进到“交互式装饰照片”：

1. 选择照片后切换操作界面；
2. 从底部 Emoji 面板选择贴纸；
3. 将贴纸覆盖到照片上；
4. 使用手势库处理拖动和双击；
5. 使用动画库实现平滑移动与尺寸切换；
6. 由开发者在实际触摸环境中验证体验。

对 React Web 开发者而言，最重要的认知变化是：移动端交互不能只验证状态和最终画面，还必须验证手势识别、连续坐标变化与动画反馈。
