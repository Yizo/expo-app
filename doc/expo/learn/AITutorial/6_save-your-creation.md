# 保存带贴纸的照片：Expo 跨平台实现思路

> 原文标题：Save your creation  
> 文档更新时间：2026 年 6 月 10 日

## 文档解决的问题

本章为 StickerSmash 应用添加最后一个核心功能：

1. 将照片和贴纸合成为一张图片。
2. 在手机端保存到系统照片库。
3. 可选地支持在浏览器中下载图片。

最终用户可以选择照片、添加贴纸，然后保存带有贴纸的成品。

## 适用场景

适合以下需求：

- 保存某个 React Native 视图的渲染结果。
- 将照片、贴纸等多个界面元素合成为图片。
- 请求手机照片库权限并写入图片。
- 同一个 Expo 项目同时支持移动端和 Web。
- 针对不同平台采用少量不同实现。

当前文档聚焦于通过 AI Agent 完成功能，没有提供完整源代码或底层 API 参数说明。

## 阅读前需要理解的概念

### Expo 项目的多平台特性

同一个 Expo 项目可以运行在：

- iOS
- Android
- Web 浏览器

这不代表所有第三方库都能跨平台运行。项目可以共享大部分代码，但在平台能力不同的位置需要增加特定分支。

对 React Web 开发者而言，可以将它理解为：

```text
共享业务逻辑和界面
        +
根据运行环境选择不同实现
```

这里的关键差异是：

| 平台 | 保存方式 |
|---|---|
| iOS / Android | 捕获指定视图，再保存到系统照片库 |
| Web | 将 DOM 转换成图片，再触发浏览器文件下载 |

### “捕获视图”是什么意思

这里不是对整个手机屏幕截图，而是将应用中指定的界面区域渲染成图片。

本章要求捕获的区域只包括：

- 用户选择的照片
- 叠加在照片上的贴纸

不应包括：

- 操作按钮
- Save 选项
- 页面其他区域
- 整个设备屏幕

贴纸会被“烘焙”进最终图片，即成为图片像素的一部分，而不是继续作为一个可单独移动的界面组件。

### 手机照片库与浏览器下载的区别

在手机端，应用会写入系统照片库。这涉及受保护的系统能力，因此第一次使用时可能需要获得用户授权。

在 Web 端，文档要求将图片作为文件下载，不是写入手机或电脑的系统照片库。

## 手机端：保存到照片库

### 交给 AI Agent 的需求

原文要求向 Agent 描述以下功能：

- 在选项列表中添加 `Save`。
- 为该选项添加下载图标。
- 点击后仅捕获照片及其贴纸。
- 使用 `react-native-view-shot` 捕获目标视图。
- 使用 `expo-media-library` 保存到手机照片库。
- 第一次需要时请求权限。
- 保存完成后显示确认提示。

### 关键依赖

#### `react-native-view-shot`

作用：将指定的 React Native 视图捕获为图片。

它负责生成带贴纸的合成图片，但不负责将图片写入系统照片库。

#### `expo-media-library`

作用：访问设备的媒体库，包括将图片保存到系统照片中。

由于照片库属于系统资源，应用可能需要先请求用户权限。

### 完整流程

```text
用户点击 Save
    ↓
检查或请求照片库权限
    ↓
用户允许访问
    ↓
捕获“照片 + 贴纸”区域
    ↓
生成合成图片
    ↓
保存到系统照片库
    ↓
显示保存成功提示
```

### 预期结果

第一次点击 `Save` 时：

1. 系统询问是否允许应用保存照片。
2. 用户点击 `Allow`。
3. 应用完成保存并显示确认提示。
4. 用户可以在手机照片库中找到带贴纸的成品。

保存后的图片可以通过其他应用继续分享。

## Web 端：下载合成图片（可选）

### 启动 Web 版本

在运行开发服务器的终端窗口中按：

```text
W
```

项目会在浏览器中打开。

该操作不是安装依赖或构建生产版本的命令，只是从当前 Expo 开发服务器启动 Web 版本。

### 为什么手机端方案不能直接复用

文档明确说明：

> `react-native-view-shot` 不支持 Web。

因此，即使应用的其他功能能够在浏览器中运行，原有 `Save` 功能也不会正常工作。

这是 React Web 开发者容易误解的一点：Expo 支持多平台，不等于项目中的每一个依赖都支持所有平台。

### Web 端使用 `dom-to-image`

文档要求在 Web 平台改用 `dom-to-image`：

1. 判断当前运行平台是否为 Web。
2. 使用 `dom-to-image` 捕获照片和贴纸所在区域。
3. 将结果作为图片文件下载。

`dom-to-image` 面向浏览器 DOM，而 `react-native-view-shot` 面向移动端 React Native 视图。两者解决的目标类似，但运行环境和处理对象不同。

### Web 保存流程

```text
用户点击 Save
    ↓
识别当前平台为 Web
    ↓
使用 dom-to-image 捕获目标区域
    ↓
生成图片
    ↓
触发浏览器文件下载
```

### 预期结果

在浏览器中：

1. 选择照片。
2. 添加贴纸。
3. 点击 `Save`。
4. 浏览器下载一张包含照片和贴纸的图片文件。

## 跨平台结构的核心思路

本章展示了 Expo 应用处理平台差异的基本方式：

```text
一个应用
+ 大部分共享代码
+ 少量平台专用分支
```

保存功能在概念上保持一致：

> 将照片和贴纸合成为可以带走的图片。

但各平台的底层实现不同：

| 环节 | 移动端 | Web |
|---|---|---|
| 捕获对象 | React Native 视图 | 浏览器 DOM |
| 捕获工具 | `react-native-view-shot` | `dom-to-image` |
| 输出方式 | 保存到系统照片库 | 下载为文件 |
| 权限处理 | 可能需要照片库权限 | 当前文档未提及权限请求 |

## 注意事项与限制

### 必须捕获正确的区域

需求明确要求只捕获照片和贴纸，不能捕获整个屏幕。

这意味着捕获目标必须对应“照片和贴纸的组合容器”。如果引用了过大的外层视图，按钮及其他界面元素可能出现在保存结果中。

具体如何绑定或引用该容器，当前文档未涉及。

### 权限可能被拒绝

文档要求第一次需要时请求权限，但没有说明：

- 用户拒绝权限后如何提示。
- 如何再次请求权限。
- 如何引导用户前往系统设置。
- iOS 与 Android 的权限差异。

因此不能仅根据本章确定完整的权限异常处理方案。

### 两个捕获库不能视为完全等价

文档只说明它们分别用于移动端和 Web，没有说明：

- 输出格式和清晰度配置。
- 图片尺寸与压缩质量。
- 复杂样式的兼容性。
- 字体、远程图片或跨域资源的处理。
- 透明背景的处理方式。

这些内容在实际开发中可能影响输出，但当前文档未涉及。

### Web 端是下载，不是写入照片库

Web 端的 `Save` 会触发文件下载。它与手机端保存到系统照片库并不是同一种系统行为，只是在用户体验上都实现了“保存成品”。

### AI Agent 的结果可能需要继续修正

如果生成结果与预期不一致，原文建议把实际看到的现象告诉 Agent，再继续调整。

这说明本章提供的是面向 Agent 的需求描述与验收结果，不保证 Agent 第一次生成的实现一定正确。

## React Web 开发者容易误解的地方

### React Native 视图不是 DOM

React Web 中通常操作 HTML 元素和 DOM；React Native 的组件最终映射到原生界面，而不是浏览器 DOM。

因此：

- Web 使用 `dom-to-image`。
- 移动端使用 `react-native-view-shot`。
- 不能因为组件写法看起来相似，就假设底层截图工具可以通用。

### 浏览器能力与原生设备能力不同

Web 下载通常由浏览器管理；移动端保存照片需要调用操作系统提供的媒体库能力，并可能触发权限弹窗。

这类似于 Web 中调用摄像头或定位前需要授权，但具体 API 和权限模型并不相同。

### “跨平台”不等于“零平台判断”

本章明确展示了平台专用分支。Expo 的价值是共享大部分项目代码，而不是消除所有平台差异。

### 保存的是合成结果，不是应用状态

最终保存的是一张包含贴纸的图片。文档没有要求保存贴纸的位置、编辑历史或可恢复的项目数据。

用户之后看到的是不可拆分的成品图片，而不是可以重新编辑的工程文件。

## 实际开发中的使用方式

以下建议均为**基于文档内容推导**：

1. 将照片和贴纸放在一个独立容器中，便于精确捕获目标区域。
2. 保持统一的 `Save` 用户入口，在内部根据平台切换实现。
3. 移动端先处理照片库权限，再执行捕获和保存。
4. Web 端避免调用 `react-native-view-shot`，改用 `dom-to-image`。
5. 分别在真实移动端和浏览器中验证，因为一个平台成功不能证明另一个平台也正常。
6. 验收时打开实际保存的文件，确认贴纸已合并，并且没有包含按钮等无关界面。

以下属于**基于经验建议**，不是当前文档明确要求：

- 对权限拒绝、捕获失败和保存失败分别提供错误提示。
- 保存过程中禁用重复点击，避免生成多份文件。
- 检查高分辨率照片下的内存消耗和输出清晰度。
- 分别测试 iOS、Android 和多个主流浏览器。
- 明确保存文件的格式、名称、尺寸和压缩质量。

## 总结

本章完成了 StickerSmash 的核心闭环：

```text
选择照片 → 添加贴纸 → 生成合成图片 → 保存或下载
```

文档明确采用两套平台实现：

- 手机端：`react-native-view-shot` 捕获视图，`expo-media-library` 保存到照片库。
- Web 端：`dom-to-image` 捕获 DOM，并通过浏览器下载文件。

对 React Web 开发者而言，最重要的认识是：Expo 项目可以共享大部分代码，但涉及原生系统能力或平台不兼容的第三方库时，仍然需要使用小范围的平台专用分支。
