# 使用 AI Agent 构建 Expo 应用：教程导读

> 原文档更新时间：2026 年 6 月 10 日

## 文档解决的问题

这篇文档是整个教程的入口，介绍如何通过指挥 AI 编程 Agent，构建一个同时运行于以下平台的 Expo 应用：

- Android
- iOS
- Web

学习者不需要亲自编写代码。AI Agent 负责实现程序，学习者负责：

- 描述产品需求
- 在手机上检查运行结果
- 发现问题
- 通过补充提示让 Agent 修改实现

需要注意：当前页面只是**教程导读**，主要说明学习目标、协作方式、准备条件和后续章节安排，并未讲解具体代码或工程配置。

## 适用场景与目标读者

### 文档明确说明

这套教程面向有应用创意、但没有编程背景的“构建者”（builders）。

适合以下场景：

- 希望通过自然语言完成一个移动应用
- 希望使用 Claude Code、Codex 或 Cursor 等 AI 编程 Agent
- 希望在真实 Android 或 iOS 手机上持续验证结果
- 希望用同一个 Expo 项目覆盖 Android、iOS 和 Web

如果目标是学习如何亲自编写代码，原文建议改用普通 Expo 教程。普通教程会构建相同的应用，但采用逐段编写代码的方式。

### 对 React Web 开发者的意义

你已经具备 React Web 开发经验，因此并不是原文所说的“零编程背景”用户。不过，这套教程仍可用于了解一种不同的开发模式：

- 你不再直接实现每个组件，而是向 Agent 描述预期结果。
- 验证环境从浏览器扩展到真实手机。
- 验收内容不仅包括界面，还包括触摸手势和系统相册访问等移动端行为。

> **基于文档内容推导：** 对 React Web 开发者而言，这套教程更适合作为 Expo 与 AI Agent 工作流的入门，而不是 React 基础教程。

## 最终要构建的应用

教程将构建一个名为 **StickerSmash** 的应用，支持：

1. 选择一张照片。
2. 在照片上放置 Emoji 贴纸。
3. 用手指移动贴纸。
4. 用手指调整贴纸大小。
5. 将最终结果保存到设备的照片库。

这些功能覆盖了几类典型能力：

| 功能 | 对 React Web 开发者的理解方式 |
|---|---|
| 选择照片 | 类似 Web 中的文件或图片选择，但实际运行在移动设备上 |
| 放置贴纸 | 类似在图片上叠加一个可交互元素 |
| 手指移动和缩放 | 属于触摸手势交互，不应只按鼠标事件理解 |
| 保存到照片库 | 涉及设备系统能力，不等同于浏览器下载文件 |
| 多平台运行 | 同一个 Expo 应用面向 Android、iOS 和 Web |

当前页面没有解释这些能力的具体 API、权限要求或实现方式，相关内容需要在后续章节中学习。

## 教程的核心协作模式

每一章都采用相同的四步循环。

### 1. Prompt：向 Agent 提出需求

将教程提供的提示词交给 AI Agent。

这里的重点不是指定每一行代码，而是描述想要的结果，例如界面应该呈现什么、用户应该能够执行什么操作。

### 2. Build：由 Agent 编写代码

Agent 根据提示修改项目代码。应用会在手机上更新，原文描述这一过程通常可以在数秒内看到结果。

当前页面没有说明：

- Agent 会修改哪些文件
- 应用通过什么机制更新
- 是否需要手动重新构建
- 出现编译错误时如何处理

这些细节不能从当前页面得出。

### 3. Verify：在手机上验证

直接在自己的手机上检查功能是否符合要求。

验证不应只停留在“代码是否生成”，还要检查：

- 页面是否正常显示
- 操作是否符合预期
- 触摸交互是否可用
- 最终结果是否满足需求

### 4. Re-prompt：描述问题并要求修正

如果结果不正确，不需要立即自己修改代码，而是向 Agent 描述实际看到的问题，让 Agent继续调整。

有效反馈应围绕可观察现象，例如：

- 哪个元素位置不正确
- 哪个按钮没有反应
- 哪个操作结果与预期不同
- 实际显示内容是什么

原文将双方职责概括为：

| 角色 | 职责 |
|---|---|
| 学习者 | 产品负责人和测试者 |
| AI Agent | 程序员 |

这意味着“不亲自写代码”并不等于“不参与开发”。学习者仍然需要明确需求、判断结果并推动问题修复。

## AI 生成结果的不确定性

### 文档明确说明

AI Agent 每次生成的结果可能不同，因此你的应用不一定与教程截图完全一致。这属于正常现象。

教程中的两类材料作用不同：

- **提示词**：描述希望实现的结果。
- **截图**：展示大致应该看到的效果。

截图不是必须逐像素复现的标准。学习者可以主动修改提示词，例如：

- 使用不同颜色
- 修改按钮文字
- 按照个人偏好调整界面

### 开发影响

验收时应优先检查功能目标，而不是机械比较截图：

- 用户是否能够完成目标操作
- 交互是否连贯
- 应用是否在目标设备上正常工作
- Agent 是否正确理解了需求

> **基于文档内容推导：** 当实现偏离预期时，应该先提供更明确的需求或现象描述，而不是默认 Agent 会在下一次生成中自动得到完全相同的结果。

## React Web 开发者容易误解的地方

### “同时运行于三端”不等于三个独立项目

文档描述的是构建一个能够运行于 Android、iOS 和 Web 的 Expo 应用，而不是分别编写三个应用。

不过，当前页面没有说明三端代码是否完全相同，也没有说明平台差异如何处理，因此不能据此认定所有实现都无需平台适配。

### 手机是主要验证环境之一

React Web 开发通常以浏览器和开发者工具作为主要反馈环境。这套教程明确要求准备真实 Android 或 iOS 手机，并在每一步之后直接检查应用。

不能只因为 Web 版本正常，就认定移动端触摸操作和设备能力也正常。

### 触摸交互不能简单等同于鼠标交互

移动和缩放贴纸需要使用手指操作。当前页面没有介绍具体手势 API，但已经表明应用包含移动端触摸交互。

### 保存到照片库不同于浏览器下载

照片库是设备系统资源。当前页面只说明应用最终能够保存结果，没有说明权限申请、失败处理或平台差异。

### AI Agent 生成代码不代表功能已经完成

教程将“构建”和“验证”分成两个独立步骤。Agent 完成代码修改后，学习者仍需在手机上验收，并在发现问题后重新提示。

## 实际学习与开发建议

### 按照每章的闭环执行

每完成一个提示后，立即执行“构建、验证、重新提示”的完整循环。不要连续提交大量需求后才统一检查，否则很难判断是哪次修改引入了问题。

> **基于文档内容推导**

### 用结果描述问题

发现异常时，应告诉 Agent“实际发生了什么”和“期望发生什么”，而不只是说“有问题”。

> **基于经验建议**

### 优先验证移动端关键行为

StickerSmash 的核心能力包括手指移动、缩放和保存到照片库，因此即使 Web 页面显示正常，也需要在手机上实际测试这些行为。

> **基于文档内容推导**

### 接受视觉差异，但不要忽略功能差异

与截图存在颜色、间距或文字差异不一定是错误；无法选择照片、无法操作贴纸或无法保存结果则属于功能问题。

> **基于文档内容推导**
