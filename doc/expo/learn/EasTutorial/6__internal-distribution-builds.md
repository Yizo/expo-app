# 创建并分享内部发布构建（Internal Distribution Build）

> 原文标题：Create and share internal distribution build  
> 文档更新时间：2026 年 5 月 18 日  
> 适用技术：Expo、EAS Build、Android、iOS

## 文档解决的问题

这篇教程介绍如何使用 **EAS Build** 创建一个可直接分享给团队成员安装的应用构建，即 **内部发布构建**。

它主要解决以下问题：

- 如何把尚未发布到应用商店的应用交给团队成员测试。
- 如何让技术人员和非技术人员直接安装应用并提供反馈。
- 如何避免测试人员必须启动 Metro 等开发服务器。
- 如何通过链接、二维码或 Expo Orbit 安装构建产物。
- 如何让开发版和预览版同时安装在同一台设备上。

内部发布构建适合团队测试、产品验收、设计走查和演示，不是面向应用商店用户的正式发布构建。

---

## 阅读前需要理解的背景知识

### React Web 项目与移动端构建的区别

React Web 项目通常将 JavaScript、CSS 等资源构建成静态文件，再由浏览器加载。

React Native/Expo 应用除了 JavaScript 代码，还必须生成能够被 Android 或 iOS 安装的原生应用文件：

| 平台 | 内部安装文件 | 类似 Web 开发中的概念 |
|---|---|---|
| Android | `.apk` | 可直接安装的软件包 |
| iOS | `.ipa` | iOS 应用安装包，但安装权限受 Apple 严格限制 |

运行 `eas build` 时，EAS Build 会在云端完成原生工程编译、应用签名和安装包生成。

### EAS Build

**EAS Build** 是 Expo 提供的云端构建服务。它可以：

- 编译 Android 和 iOS 应用；
- 处理应用签名凭据；
- 保存构建产物；
- 生成可分享的构建详情链接；
- 提供安装说明。

它不是 React Web 中单纯执行 `npm run build` 的同类工具，因为移动应用构建还涉及平台证书、签名和设备安装权限。

### 开发构建与内部发布构建

| 特性 | 开发构建 | 内部发布构建 |
|---|---|---|
| 主要使用者 | 开发人员 | 开发、测试、产品、设计等团队成员 |
| 是否需要开发服务器 | 通常需要 | 不需要 |
| 是否适合非技术人员 | 通常不适合 | 适合 |
| 主要目的 | 开发和调试 | 独立运行、内部测试和反馈 |

内部发布构建安装后可以直接点击图标启动，不需要测试人员在电脑上运行开发服务器。

## 内部分发方式

Android 和 Apple 本身提供了应用内部测试机制：

- Android：Google Play Beta
- iOS：TestFlight

原文指出，这些传统方式存在一定限制，并以 **TestFlight 同一时间只能有一个活跃构建**为例。

EAS Build 提供了另一种更直接的方式：

1. 在云端生成可安装文件；
2. 为构建生成可分享链接；
3. 测试人员通过链接查看安装方式；
4. 将应用直接安装到允许的设备上。

这使它更适合需要快速向团队提供测试版本的场景。

## 创建内部发布构建

### 1. 配置 `preview` 构建档案

EAS 的构建配置位于项目根目录下的 `eas.json`。

初始配置已经提供了用于内部分发的 `preview` profile：

```json
{
  "build": {
    "preview": {
      "distribution": "internal"
    }
  }
}
```

这里的配置含义如下：

| 配置 | 含义 |
|---|---|
| `build` | 所有 EAS 构建档案的集合 |
| `preview` | 当前构建档案的名称，可以通过命令行选择 |
| `distribution` | 指定应用的发布方式 |
| `"internal"` | 生成供内部设备直接安装的构建 |

将 `distribution` 设置为 `internal` 后：

- 构建链接可以分享给其他人；
- 安装后的应用不依赖开发服务器；
- Android 会生成适合设备安装的 `.apk`；
- iOS 会生成适合设备安装的 `.ipa`。

`preview` 只是 profile 的名称，真正决定内部分发行为的是：

```json
"distribution": "internal"
```

### 2. 准备应用签名凭据

创建内部发布构建需要 **应用签名凭据（app signing credentials）**。

应用签名可以理解为移动应用的数字身份。操作系统会通过签名确认应用来源，并判断后续版本能否作为同一个应用进行安装或升级。

#### Android 签名

Android 对内部安装的限制相对宽松，任何兼容设备都可以安装有效签名的 `.apk`。

原文教程此前创建开发构建时已经生成了 Android Keystore，因此预览构建不需要再创建新的 Keystore。

**Keystore** 是保存 Android 应用签名密钥的文件。它对应用身份和后续更新非常重要。

> 当前文档只说明可以复用之前生成的 Keystore，没有展开介绍 Keystore 的备份、迁移或丢失处理方式。

#### iOS 签名

Apple 对绕过 App Store 的设备安装限制更严格。

普通 iOS 内部分发需要使用 **Ad Hoc Provisioning Profile**。该描述文件必须明确列出允许运行应用的设备。

这意味着：

- 不是拿到 `.ipa` 文件的所有人都一定能安装；
- 测试设备必须被加入相应的描述文件；
- 新增测试设备后，可能需要更新描述文件并重新构建。

部分满足 Apple 特定要求的组织，可以通过 **Apple Developer Enterprise Program** 面向更大规模的内部用户分发应用。

> 当前文档未说明企业开发者计划的申请条件和具体配置过程。

### 3. 发起 Android 构建

使用 `preview` profile 创建 Android 内部发布构建：

```sh
eas build --platform android --profile preview
```

参数说明：

| 参数 | 作用 |
|---|---|
| `eas build` | 启动 EAS 云端构建 |
| `--platform android` | 指定构建 Android 应用 |
| `--profile preview` | 使用 `eas.json` 中的 `preview` 配置 |

命令提交后，可以在 EAS Dashboard 中查看构建进度。

原文明确展示的是 Android 命令，没有单独给出 iOS 构建命令；但总结说明本章同时创建了 Android 和 iOS 内部发布构建。

## 安装构建产物

构建完成后，构建详情页中的 **Build artifact** 区域会显示构建产物及可用安装方式。

文档介绍了两种安装方案。

### 方案一：使用 Expo Orbit

**Expo Orbit** 是运行在开发者电脑上的桌面工具，可以帮助把 Expo 构建产物安装到连接的移动设备。

操作步骤：

1. 打开构建详情页。
2. 如需让其他成员访问，可以把构建页面链接发送给对方。
3. 使用 USB 将 Android 或 iOS 设备连接到电脑。
4. 打开 Expo Orbit 菜单栏应用。
5. 在 Orbit 中选择目标设备。
6. 在构建详情页的 **Build artifact** 区域点击 **Open with Orbit**。

这种方式需要：

- 电脑已安装 Expo Orbit；
- 手机能够通过 USB 连接到电脑；
- Orbit 能识别目标设备。

### 方案二：使用安装链接或二维码

不使用 Orbit 时，可以从构建详情页获取安装链接：

1. 打开构建详情页。
2. 在 **Build artifact** 区域点击 **Install**。
3. 页面会显示 **Install on a test device** 弹窗。
4. 从 **Send a link to a device** 区域复制链接。
5. 将链接发送到测试设备，并在设备上打开。

原文将此方案称为“Install and QR code”，但给出的文字步骤主要描述了复制和发送链接，没有进一步说明二维码操作细节。

## 运行预览构建

安装完成后，直接点击设备上的应用图标即可启动。

```text
不需要启动开发服务器
```

这对非技术测试人员非常重要：他们不需要安装 Node.js、运行 Expo CLI，也不需要让手机连接到开发者电脑上的 Metro 服务。

如果项目已经配置了多个应用变体，开发构建和预览构建可以作为两个独立应用同时安装在同一台设备上。

例如：

- 开发人员使用开发变体连接开发服务器调试；
- 测试人员使用预览变体测试一个独立、稳定的构建；
- 两者不会因为应用身份相同而相互覆盖。

> 原文说明此前章节已经完成多应用变体配置，但当前文档没有介绍具体配置方法。

## 完整流程

```text
在 eas.json 中配置 preview profile
              ↓
设置 distribution 为 internal
              ↓
准备 Android/iOS 应用签名凭据
              ↓
运行 eas build 并选择 preview profile
              ↓
EAS 云端生成 .apk 或 .ipa
              ↓
打开构建详情页
              ↓
通过 Orbit、安装链接或二维码安装
              ↓
直接点击应用图标运行，无需开发服务器
```

## 注意事项、限制条件和坑点

### iOS 链接可分享不等于任何设备都能安装

`distribution: "internal"` 会生成可分享链接，但 iOS 的 Ad Hoc 分发仍然受设备名单限制。

因此必须区分：

- **有权打开构建页面**
- **设备有权安装并运行该应用**

这两个条件并不等价。

### Android 和 iOS 的内部安装机制不同

Android 的 `.apk` 通常可以直接安装到兼容设备上，而 iOS 的 `.ipa` 受到签名、描述文件和设备注册限制。

不能因为 Android 分享链接可以直接使用，就推断 iOS 也具有完全相同的安装体验。

### 内部发布构建不是开发构建

虽然二者都不是应用商店正式版本，但用途不同：

- 开发构建服务于代码调试，需要开发环境配合；
- 内部发布构建服务于独立测试，不需要开发服务器。

原文安装章节有一句使用了“running the development build”的表述，但该章节上下文讨论的是内部发布构建。按上下文理解，这里指的是运行当前生成的构建产物。

### 多个变体同时安装需要前置配置

文档展示了开发版和预览版同时存在的结果，但明确说明这是因为教程此前已经设置了多个应用变体。

仅仅创建两个 EAS profile，并不一定会让系统把它们识别为两个独立应用。当前文档没有提供多变体配置细节。

- 如何发布到 App Store 或 Google Play；

## React Web 开发者最容易误解的地方

### “构建成功”不代表所有手机都能安装

Web 构建产物通常只要能部署到服务器，用户就可以通过浏览器访问。

移动应用还必须满足：

- 安装包格式正确；
- 应用签名有效；
- 操作系统允许该分发方式；
- iOS 设备在描述文件允许范围内。

### 构建 profile 不只是环境变量集合

在 Web 项目中，`development`、`staging`、`production` 往往主要代表不同环境变量。

EAS profile 还可能决定：

- 应用如何分发；
- 生成什么类型的安装包；
- 使用什么签名方式；
- 构建产物能否直接安装。

本教程中，`preview` profile 的核心作用就是通过 `distribution: "internal"` 指定内部分发。

### “不需要开发服务器”意味着应用能够独立运行

开发构建可能从 Metro 开发服务器加载 JavaScript。

内部发布构建已经包含运行所需的应用代码和资源，因此测试人员不需要保持开发者电脑在线。这更接近把 Web 应用完整部署到测试环境后的使用体验。

### 分享链接不是公开部署一个 Web 地址

构建链接主要用于访问构建详情和安装产物。用户最终运行的是安装在设备上的原生应用，而不是在浏览器中打开的网页。

## 实际开发中的使用方式

下面是根据文档流程整理出的典型使用场景：

1. 开发人员完成一个准备交付测试的功能版本。
2. 使用 `preview` profile 创建内部发布构建。
3. 在 EAS Dashboard 中等待构建完成。
4. 将构建详情链接发送给测试、产品或设计人员。
5. Android 测试人员安装 `.apk`。
6. iOS 测试人员在设备满足 Ad Hoc 描述文件要求的前提下安装 `.ipa`。
7. 测试人员直接启动应用并反馈问题。
8. 开发人员仍可保留开发构建进行日常调试。

### 基于文档内容推导

由于内部发布构建不依赖开发服务器，它比开发构建更适合：

- 异步远程测试；
- 向非技术成员演示；
- 验证应用冷启动和独立运行行为；
- 在提交应用商店前进行团队验收。

### 基于经验建议

- 将 `preview` 作为稳定的内部验收环境，不要让它与日常调试配置完全混用。
- 在分享 iOS 构建前，先确认测试设备已加入允许列表，避免对方拿到链接却无法安装。
- 在构建说明中记录代码版本、测试范围和后端环境，避免团队成员测试错版本。
- 谨慎保管 Android Keystore 和 Apple 签名凭据，它们关系到应用身份和后续更新能力。

以上建议是工程实践补充，并非当前文档明确规定。

## 总结

这篇文档的核心配置是：

```json
{
  "build": {
    "preview": {
      "distribution": "internal"
    }
  }
}
```

使用该 profile 执行 EAS Build 后，可以生成无需开发服务器即可运行的内部测试安装包，并通过 EAS 构建页面、Expo Orbit、安装链接或二维码分享。

Android 内部分发相对直接；iOS 则需要重点处理应用签名和 Ad Hoc 设备授权。对于 React Web 开发者，最关键的认知变化是：移动端交付不仅是编译代码，还包括生成平台安装包、签名应用以及满足设备安装权限要求。
