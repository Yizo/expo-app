# Expo SMS 学习文档

## 文档解决的问题

`expo-sms` 是 Expo 提供的短信调用库，用于从 React Native 应用中打开设备的系统短信界面，并预填以下内容：

- 一个或多个收件人手机号
- 短信正文
- 可选附件

用户随后在系统短信应用中确认发送、取消发送，或者进行其他操作。

需要特别明确：`expo-sms` **不是后台短信发送服务**。它不会绕过用户直接发送短信，而是把用户带到系统提供的短信界面。

本文对应 **下一版本 Expo SDK 的未版本化文档**，修改日期为 **2026 年 1 月 15 日**。原文提示，当前最新稳定文档为 **SDK 56**。实际项目应优先查看与项目 Expo SDK 版本匹配的文档。

## 适用平台与场景

文档明确支持：

- Android
- iOS
- Expo Go

适合以下场景：

- “联系我们”页面预填客服号码和咨询内容
- 邀请用户通过短信分享内容
- 将订单号、验证码或其他文本带入短信编辑器
- 让用户选择是否发送预先准备好的短信
- 在系统短信界面中添加受支持的附件

不适合以下需求：

- 在后台自动发送短信
- 不经过用户确认直接发送短信
- 精确确认短信是否成功送达收件人
- 在浏览器中发送短信
- 使用 iOS 模拟器测试真实短信能力

其中，后台发送和送达状态等限制可以从 API 的行为和返回值推导出来，但原文没有讨论短信服务商、服务器短信接口或送达回执的替代方案。

## React Web 开发者需要理解的背景

### 系统短信界面是什么

在 React Web 中，页面通常只能尝试通过 `sms:` URL 调起外部应用，浏览器无法直接控制系统短信应用。

在 React Native 和 Expo 中，`expo-sms` 通过原生能力打开 Android 或 iOS 的默认短信界面。JavaScript 代码负责提供初始数据，但真正的发送操作由系统应用和用户完成。

可以将其理解为：

1. React Native 应用准备收件人、正文和附件。
2. `expo-sms` 把这些数据交给操作系统。
3. 操作系统打开短信应用。
4. 用户检查并决定是否发送。
5. 系统关闭短信界面后，模块尽可能返回操作结果。

### Expo Go 是什么

Expo Go 是用于运行和调试 Expo 项目的移动端应用。文档标记 `expo-sms` 为 “Included in Expo Go”，表示 Expo Go 已包含该模块对应的原生实现。

因此，在普通 Expo 项目中安装 JavaScript 包后，可以直接通过 Expo Go 调试，不需要为了这个模块单独构建原生客户端。

### Promise 返回值不等于发送成功

两个主要 API 都是异步方法，返回 `Promise`。但 `Promise` 成功兑现只表示调用流程正常完成，并不一定表示短信已经成功送达。

特别是在 Android 上，调用 `sendSMSAsync()` 后结果始终是 `unknown`。因此不能把下面的代码理解为“短信发送成功”：

```ts
const response = await SMS.sendSMSAsync(phoneNumber, message);

// Android 上这里可能正常执行，但 response.result 仍然是 'unknown'
```

## 安装

根据项目使用的包管理器执行对应命令：

```sh
# npm
npx expo install expo-sms

# yarn
yarn expo install expo-sms

# pnpm
pnpm expo install expo-sms

# bun
bun expo install expo-sms
```

`expo install` 与直接执行 `npm install` 的关注点不同：它会根据当前 Expo SDK 选择兼容的包版本。对于 Expo 项目，应优先使用文档提供的安装命令。

如果是在已有的普通 React Native 工程中使用该库，也就是并非由 Expo 项目模板创建的工程，需要先按照 Expo 文档将 `expo` 和 Expo Modules 支持安装到项目中。仅安装 `expo-sms` 包可能不足以让其原生部分正常工作。

当前文档没有涉及：

- Android 原生工程的手动配置
- iOS Pod 安装步骤
- 权限声明配置
- `app.json` 或 `app.config.js` 配置
- Config Plugin 配置
- EAS Build 配置

不要在没有其他依据的情况下自行添加文档未要求的短信权限。

## 引入模块

```ts
import * as SMS from 'expo-sms';
```

这里使用命名空间导入，之后通过 `SMS.isAvailableAsync()`、`SMS.sendSMSAsync()` 等形式调用 API。

## 推荐调用流程

完整的基本流程是：

1. 检查当前设备是否支持短信功能。
2. 如果支持，打开系统短信界面。
3. 等待用户完成或取消操作。
4. 根据平台能力处理返回结果。

```ts
import * as SMS from 'expo-sms';

async function composeSMS() {
  const isAvailable = await SMS.isAvailableAsync();

  if (!isAvailable) {
    // 在界面中提示当前设备无法使用短信功能
    return;
  }

  const { result } = await SMS.sendSMSAsync(
    '0123456789',
    '这是一条预填的短信内容'
  );

  if (result === 'sent') {
    // 系统报告用户已经发送或安排发送
  } else if (result === 'cancelled') {
    // 用户取消了短信操作
  } else {
    // 平台无法确定状态，Android 会进入这个分支
  }
}
```

检查可用性不是形式步骤。浏览器和 iOS 模拟器中短信不可用，部分真实设备也可能因为自身能力或环境无法使用短信功能。

## API 详解

### `SMS.isAvailableAsync()`

```ts
const isAvailable = await SMS.isAvailableAsync();
```

支持平台：

- Android
- iOS

返回类型：

```ts
Promise<boolean>
```

该方法判断当前设备是否可以使用短信功能：

- `true`：可以继续调用短信编辑接口。
- `false`：当前环境不能使用短信功能。

文档明确说明：

- 在 iOS 模拟器中始终返回 `false`。
- 在浏览器中始终返回 `false`。

示例：

```ts
const isAvailable = await SMS.isAvailableAsync();

if (isAvailable) {
  // 调用短信功能
} else {
  // 显示短信功能不可用的替代界面
}
```

对 React Web 开发者而言，“代码可以加载”与“设备具备短信能力”是两件事。移动端功能经常依赖真实设备硬件、系统应用和当前运行环境，因此应先进行运行时能力检测。

### `SMS.sendSMSAsync(addresses, message, options)`

```ts
const response = await SMS.sendSMSAsync(addresses, message, options);
```

支持平台：

- Android
- iOS

该方法打开系统默认短信界面，并预填收件人、正文以及可选附件。

它不会直接在后台发出短信。用户仍然需要在系统界面中决定是否发送。

#### `addresses`

类型：

```ts
string | string[]
```

可以传入一个手机号：

```ts
await SMS.sendSMSAsync('0123456789', 'Hello');
```

也可以传入多个手机号：

```ts
await SMS.sendSMSAsync(
  ['0123456789', '9876543210'],
  'Hello'
);
```

这些号码会作为预填收件人显示在短信界面中。

文档未说明：

- 手机号格式校验规则
- 是否必须包含国家或地区代码
- 无效号码的具体处理方式
- 不同平台对多收件人的行为差异

因此，业务代码不应依据这篇文档假设任意字符串都一定能被系统短信应用接受。

#### `message`

类型：

```ts
string
```

表示预填的短信正文：

```ts
await SMS.sendSMSAsync(
  '0123456789',
  'My sample HelloWorld message'
);
```

文档没有说明短信长度、字符编码、分段短信或运营商计费规则。

#### `options`

类型：

```ts
SMSOptions
```

可选参数，用于传递额外的短信配置。当前文档只列出了附件配置：

```ts
{
  attachments: {
    uri: 'path/myfile.png',
    mimeType: 'image/png',
    filename: 'myfile.png',
  },
}
```

`attachments` 的类型为：

```ts
SMSAttachment | SMSAttachment[]
```

因此可以表示一个附件或多个附件。不过原文没有进一步说明不同平台是否都支持多个附件，也没有说明附件大小和格式限制。

#### 返回结果

返回类型：

```ts
Promise<SMSResponse>
```

`SMSResponse` 的结构为：

```ts
{
  result: 'unknown' | 'sent' | 'cancelled';
}
```

各状态含义如下：

| 状态 | 含义 |
| --- | --- |
| `sent` | 用户已经发送短信，或者已经安排短信发送 |
| `cancelled` | 用户取消了短信发送流程 |
| `unknown` | 系统无法确定短信操作状态 |

Android 不提供短信状态信息，因此在 Android 设备上，Promise **始终**兑现为：

```ts
{ result: 'unknown' }
```

这意味着 Android 上的 `unknown` 不是异常，也不能说明用户发送失败。

此外，文档明确说明，该模块只能获得“是否发送过任何消息”这一层面的反馈，不会检查：

- 实际发送的短信正文
- 最终收件人列表

用户进入系统短信界面后可能修改预填内容或收件人，因此应用不能把传入参数当作最终发送内容的可靠记录。

## 附件配置

### `SMSAttachment`

附件对象包含三个必填属性：

| 属性 | 类型 | 作用 |
| --- | --- | --- |
| `filename` | `string` | 附件文件名，例如 `myfile.png` |
| `mimeType` | `string` | 附件的 MIME 类型，例如 `image/png` |
| `uri` | `string` | 附件的 Content URI |

完整示例：

```ts
const attachment = {
  uri: 'content://...',
  mimeType: 'image/png',
  filename: 'myfile.png',
};

await SMS.sendSMSAsync(
  ['0123456789', '9876543210'],
  'My sample HelloWorld message',
  {
    attachments: attachment,
  }
);
```

### 为什么 `uri` 必须是 Content URI

文档明确要求附件 URI 必须是 **Content URI**，这样 Expo 应用之外的其他应用才能访问该文件。

这与 React Web 中的 Blob URL 或静态资源 URL 不同。短信编辑器是另一个原生应用，它不能默认访问当前应用的私有文件路径。Content URI 是移动端用于向其他应用暴露文件内容的一种系统 URI。

文档指向 `FileSystem.getContentUriAsync(fileUri)`，用于把文件 URI 转换成可供其他应用访问的 Content URI。

从开发影响来看，下面这种普通文件路径不应被直接视为符合要求：

```ts
{
  uri: 'path/myfile.png',
  mimeType: 'image/png',
  filename: 'myfile.png',
}
```

虽然原文调用示例使用了类似路径，但类型说明明确要求 Content URI。实际使用附件时，应以类型说明中的要求为准，并参考 Expo FileSystem 文档完成 URI 转换。

当前文档没有说明：

- 如何首先下载或创建附件文件
- `FileSystem.getContentUriAsync()` 的完整参数和调用流程
- iOS 与 Android 的附件 URI 差异
- 支持的附件类型、数量和大小上限
- 系统短信应用是否一定接受附件

## 关键限制与容易踩坑的地方

### Android 无法确认发送状态

Android 上 `result` 始终为 `unknown`。以下判断不能作为跨平台发送成功逻辑：

```ts
const { result } = await SMS.sendSMSAsync(phone, message);

if (result !== 'sent') {
  // 错误：在 Android 上不能据此认定发送失败
}
```

更合理的业务含义是：

- `sent`：系统明确报告用户执行了发送或安排发送。
- `cancelled`：系统明确报告用户取消。
- `unknown`：应用无法判断结果，应避免显示“发送失败”。

### `sent` 不是送达回执

即使返回 `sent`，文档也没有承诺消息已经到达对方设备。它描述的是用户触发的短信操作状态，而不是运营商级别的送达状态。

如果业务必须获得可靠的发送和送达记录，仅依据 `expo-sms` 不足以实现。

### 无法核实最终正文和收件人

应用传入的是预填值，用户可以在系统短信界面中修改它们。模块不会检查最终内容或最终收件人。

因此，不要将 `sendSMSAsync()` 的调用参数直接记录为“用户实际发送的短信”。

### 浏览器和 iOS 模拟器不可用

浏览器和 iOS 模拟器中的 `isAvailableAsync()` 始终返回 `false`。短信流程应使用具备短信能力的真实设备测试。

当前文档没有明确说明 Android 模拟器的具体表现，因此不能仅根据本文断言其返回值或可测试范围。

### 系统界面不受 React 组件控制

调用 API 后出现的是操作系统或默认短信应用的界面，不是 React Native 组件。因此不能像控制 React 弹窗一样：

- 自定义短信编辑器的完整样式
- 控制系统发送按钮
- 阻止用户修改内容
- 读取用户最终编辑的正文

## 实际开发建议

以下内容属于**基于文档内容推导**：

1. 在显示短信入口前调用 `isAvailableAsync()`，不可用时隐藏按钮或提供其他联系方式。
2. 将 `unknown` 设计为正常的“不确定状态”，尤其不能在 Android 上将其映射为发送失败。
3. 不要因为返回 `sent` 就触发依赖可靠送达的关键业务，例如确认付款、完成身份验证或认定通知已经送达。
4. 使用附件前，应先确保 URI 能被系统短信应用访问。
5. 需要分别在 Android 和 iOS 真实设备上验证，因为两端能够提供的操作结果不同。
6. 业务埋点可以记录“用户打开了短信界面”和模块返回的状态，但不应声称记录了实际短信内容或最终收件人。

以下属于**基于经验建议**：

- 调用异步 API 时使用 `try...catch`，处理系统调用异常或运行环境问题。
- 在用户快速连续点击时暂时禁用按钮，避免重复打开系统短信界面。
- 对手机号和正文进行业务层校验，但不要把业务校验等同于系统或运营商最终接受。
- 项目应查看与自身 Expo SDK 版本对应的文档，不要直接照搬未版本化文档中的 API。

示例：

```ts
import * as SMS from 'expo-sms';

export async function openSupportSMS() {
  try {
    const available = await SMS.isAvailableAsync();

    if (!available) {
      return { status: 'unavailable' as const };
    }

    const { result } = await SMS.sendSMSAsync(
      '0123456789',
      '你好，我需要咨询以下问题：'
    );

    return { status: result };
  } catch {
    return { status: 'error' as const };
  }
}
```

## 总结

`expo-sms` 的核心作用是打开 Android 或 iOS 的系统短信界面，并预填收件人、正文和可选附件。

使用时最重要的原则是：

- 先通过 `isAvailableAsync()` 检查设备能力。
- `sendSMSAsync()` 打开的是系统界面，不会静默发送短信。
- Android 始终返回 `unknown`，不能据此判断发送失败。
- `sent` 代表用户触发了发送或安排发送，不代表短信已经送达。
- 模块无法确认用户最终发送的正文和收件人。
- 附件必须使用其他应用可以访问的 Content URI。
- 真实短信能力应在真实移动设备上测试。
