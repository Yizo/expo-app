# 发布状态（Release Statuses）

## 文档解决的问题

在 Expo 生态中，SDK（软件开发工具包）和各种服务功能并不都处于相同的成熟度阶段。有些功能刚刚推出、还在试验阶段，有些则已经过充分验证、可以放心用于生产环境。

这篇文档定义了 Expo 使用的一套**发布状态标签**，帮助开发者快速判断某个功能或 SDK 版本当前的可靠程度，从而做出是否在生产项目中使用的决策。

**类比 Web 开发：** 这类似于 npm 包标记为 `alpha`、`beta`、`stable` 或 `deprecated`。你在选择前端依赖库时也会关注它的版本状态——对一个 `0.x` 的库和 `5.x` 的库，信任度是不同的。Expo 的发布状态本质上在做同一件事，只是分类更细致。

## 阅读前需要理解的背景知识

- **SDK（Software Development Kit）**：在 Expo 语境中，SDK 是 Expo 提供的一整套工具、API 和服务的集合。每个 SDK 版本会捆绑特定版本的 React Native 和一系列 Expo 模块。你可以把它理解为 Web 开发中的"框架版本"——比如 Next.js 14 和 Next.js 15 就是不同版本。
- **语义化版本（Semantic Versioning）**：一种版本号规范，格式为 `主版本号.次版本号.补丁版本号`（如 `3.2.1`）。主版本号变更意味着可能有破坏性改动（Breaking Changes），次版本号变更意味着向后兼容的新功能，补丁版本号变更意味着向后兼容的 Bug 修复。Expo 的 Stable 阶段功能遵循这一规范。
- **生产环境（Production）**：面向真实用户运行的环境。在 Web 开发中就是你部署上线的网站。在移动端就是用户从应用商店下载安装后的 App。

## 五种发布状态详解

Expo 定义了五种发布状态，按照功能从"最早期"到"已淘汰"排列。每个状态都有对应的标识（Badge），在 Expo 文档页面中可以看到。

### 1. Alpha（内测阶段）

Alpha 是功能最早对外发布的阶段，目的是让社区**尽早试用并提供反馈**。

**预期情况：**

| 特征 | 说明 |
|------|------|
| API 可能大幅变更 | 接口的名称、参数、返回值都可能在后续版本中发生破坏性改动，无需通过主版本号升级来通知 |
| 核心实现可能重写 | 根据社区反馈，底层架构可能被大幅修改 |
| 可能存在缺陷或性能问题 | Bug 多、运行不稳定是常态 |
| 禁止用于生产环境 | 绝对不要在面向真实用户的应用中使用 |

**核心理念：** Alpha 功能让你有机会**影响 Expo 未来的发展方向**。你可以把它当作"参与设计"的机会，在测试环境中尝试并给出意见。

**对 Web 开发者的类比：** 类似于 React 的 `experimental` 发布通道（如 `react@experimental`），里面的 API（比如早期的 `<Suspense>` 或 Server Components）可能随时变化，只适合实验性探索。

### 2. Preview（预览阶段）

Preview 让你提前看到某个新功能的**局部能力**。与 Alpha 相比，Preview 的范围更小、更聚焦，但功能覆盖面不完整。

**预期情况：**

| 特征 | 说明 |
|------|------|
| 提供功能的子集 | 不是完整实现，只展示部分能力 |
| 实现效率不错但未经全面验证 | 代码质量可以，但尚未覆盖所有生产场景 |
| API 可能继续调整 | 随着开发推进，接口仍可能变化 |
| 可用于生产环境（条件限制） | 如果你充分评估过风险并且能接受功能受限，可以在生产中使用 |

**核心理念：** Preview 的目的是**尽早分享新功能的实际效果**，在扩展功能之前先收集使用者的反馈。

**与 Alpha 的关键区别：** Preview 更聚焦、更稳定，范围更窄。Alpha 是整个功能的早期尝试，Preview 是特定切片的提前展示。

### 3. Beta（公测阶段）

Beta 阶段的功能已经**具备完整的核心能力**，正在进行正式发布前的最终验证。

**预期情况：**

| 特征 | 说明 |
|------|------|
| 核心功能已完成 | 主要能力已经实现完毕 |
| API 设计基本定型 | 接口不会再有大幅改动 |
| 可能残留小 Bug 或边界情况 | 极端场景下可能有问题 |
| 大幅变更不太可能 | 除非出现严重问题，否则不会做破坏性修改 |
| 适合生产环境（需充分测试） | 在完成全面评估后可以部署到生产应用 |

**核心理念：** Beta 功能**已准备好接受真实环境的测试**。文档建议在正式推送给终端用户之前，先做充分的测试验证。

**对 Web 开发者的类比：** 类似于 Chrome 的 Beta 通道或 Next.js 的 Release Candidate（RC）版本——核心功能已经确定，主要在做稳定性验证。

### 4. Stable（稳定阶段）

没有特殊标签的功能就是 Stable 状态，代表**已完全准备好用于生产环境**。

**预期情况：**

| 特征 | 说明 |
|------|------|
| 完全支持生产使用 | 可以放心用在面向用户的 App 中 |
| 破坏性变更仅在主版本升级时发生 | 遵循语义化版本，小版本和补丁版本不会引入 Breaking Changes |
| 可靠性和性能已确认 | 经过充分测试，稳定可用 |
| 文档和示例齐全 | 提供完整的使用指南和代码示例 |

**核心理念：** 你可以信赖 Stable 功能**遵循语义化版本原则**——破坏性改动只会在主版本号变更时出现。

**对 Web 开发者的类比：** 就是你在 `package.json` 中用的那些稳定的 npm 包，比如 `react@18.x`、`next@14.x`——有完善的版本管理策略，你可以放心依赖。

### 5. Deprecated（已弃用）

Deprecated 标记的功能**不建议继续使用**，并将在未来某个时间点被移除。Expo 会提前发出警告，给开发者留出迁移时间。

**预期情况：**

| 特征 | 说明 |
|------|------|
| 代码和文档中会显示警告 | 使用时会看到弃用提示 |
| 不再接受任何改进 | 不会修 Bug，不会加新功能 |
| 未来的主版本升级中会被移除 | 在某次大版本更新时彻底删除 |

**建议：** 遇到 Deprecated 功能时，应该**尽快迁移到文档推荐的替代方案**，以保持项目的可维护性。

**对 Web 开发者的类比：** 类似于 React 中 `componentWillMount` 等旧生命周期方法被标记为 `UNSAFE_` 并最终移除的过程。你看到弃用警告时就应该着手迁移，而不是等到下一个大版本升级时被动应对。

## 五种状态的对比总览

| 状态 | 功能完整度 | API 稳定性 | 可用于生产 | 核心目的 |
|------|-----------|-----------|-----------|---------|
| Alpha | 低 | 极不稳定 | 禁止 | 收集早期反馈，影响设计方向 |
| Preview | 部分 | 可能变化 | 有条件可以 | 提前展示新能力切片 |
| Beta | 基本完整 | 基本稳定 | 充分测试后可以 | 正式发布的最终验证 |
| Stable | 完整 | 稳定（遵循语义化版本） | 完全支持 | 生产环境的主力 |
| Deprecated | 完整（但已废弃） | 不再维护 | 不建议 | 引导迁移到替代方案 |

## 注意事项、限制条件和坑点

1. **Alpha 和 Preview 的 API 不稳定**：这意味着你基于这些 API 写的代码，在下一次 Expo SDK 升级时可能需要大幅修改甚至重写。如果你在项目中使用了这些功能，升级 SDK 时要额外留意变更日志（Changelog）。

2. **Beta 虽然可以用在生产，但"充分测试"是前提**：不要看到"可用于生产"就跳过测试。Beta 阶段仍可能有边界情况的 Bug，你需要在测试环境覆盖足够多的场景。

3. **Deprecated 功能不会立即消失**：Expo 会给出迁移窗口期，但不要因此拖延。等到主版本升级强制移除时再改，成本会更高、风险也更大。

4. **文档中的标签是动态变化的**：一个功能可能从 Alpha 推进到 Preview、再到 Beta、最终成为 Stable。在规划项目时，如果你依赖了某个 Beta 功能，需要关注它的推进进度。

5. **Stable 功能的破坏性变更只在主版本升级时发生**：这并不意味着你可以无限期不升级。当 Expo 发布新的大版本时，你需要评估升级成本和迁移工作量。

## React Web 开发者需要特别注意的地方

1. **移动端对"不稳定功能"的容忍度更低**：在 Web 开发中，如果某个实验性 API 出了问题，用户刷新页面就能恢复。但在移动端，App 崩溃意味着用户需要重新打开应用，体验损失更大。因此，**在非实验性项目中尽量避免使用 Alpha 和 Preview 功能**。

2. **移动应用的发布周期更长**：Web 应用可以随时部署修复，但移动应用需要通过 App Store / Google Play 审核（通常需要数小时到数天）。这意味着如果你在 Stable 功能之外冒险，出问题的修复成本更高。

3. **版本升级不只是改 `package.json`**：在 Web 项目中升级 Next.js 版本通常只需要修改依赖版本号。但在 Expo 项目中，SDK 版本升级可能涉及原生代码变更、设备兼容性测试等额外工作。选择 SDK 版本时要考虑其发布状态对长期维护的影响。

## 实际开发建议

- **新项目选型**：优先使用 Stable 状态的功能。如果需要某个 Beta 功能才能实现核心业务，评估其稳定性后再决定。
- **技术探索和学习**：在独立的实验项目（非生产项目）中尝试 Alpha 和 Preview 功能，给 Expo 团队提供反馈。
- **技术债务管理**：定期扫描项目中使用的 Deprecated 功能，制定迁移计划。可以在代码中用注释标记这些用法，方便后续追踪。
- **升级策略**（基于经验建议）：关注 Expo 的发布博客和 Changelog，在每次 SDK 大版本升级前，检查你所依赖的功能的发布状态是否有变化。

## 总结

Expo 的发布状态体系是一套从 Alpha 到 Deprecated 的五级分类系统，核心目的是帮助开发者判断功能的成熟度和风险等级。理解这套体系后，你可以：

- 在选型时避开不稳定的功能
- 在升级时预判哪些 API 可能发生变化
- 在遇到弃用警告时及时迁移
- 在技术探索时合理利用早期阶段的功能

这套思路与 Web 生态中的版本管理理念一脉相承，只是在移动端场景下需要更加谨慎——因为发布和修复的成本都更高。
