# Expo UI 学习笔记

## 文档解决的问题

Expo UI（包名为 `@expo/ui`）提供一组原生输入和界面组件，使开发者可以在 React 代码中使用：

- Android 的 Jetpack Compose 组件
- Apple 平台的 SwiftUI 组件
- 同时支持 Android、iOS 和 Web 的通用组件

它的目标是覆盖典型应用经常需要的功能和组件，让开发者能够构建真正使用原生 UI 技术实现的界面。

> **文档明确说明**：Expo UI 是“一组允许你从 React 直接使用 Jetpack Compose 和 SwiftUI 构建 UI 的组件”。

## 适用场景

Expo UI 适合以下需求：

- 使用 React 开发 Android 原生界面
- 使用 React 开发 iOS 或 tvOS 原生界面
- 希望同一套组件代码同时运行在 Android、iOS 和 Web
- 需要替换某些常用的 React Native 社区组件库
- 希望使用平台原生 UI 组件，而不只是构建类似原生界面的视觉效果

Expo UI 当前支持：

| 平台 | 支持情况 |
| --- | --- |
| Android | 支持 |
| iOS | 支持 |
| tvOS | 支持 |
| Web | 由 Universal 组件支持 |
| Expo Go | 已包含 |

> **注意**：原文只说明 Expo UI 包含在 Expo Go 中，没有进一步介绍 Expo Go 的使用流程或兼容限制。

## React Web 开发者需要了解的背景

### React Native 与 React Web 的区别

React Web 最终通常通过 `react-dom` 创建浏览器 DOM，例如：

```jsx
<button>提交</button>
```

浏览器负责渲染对应的 HTML 元素。

React Native 不以浏览器 DOM 为主要渲染目标。组件需要映射到 Android、iOS 等平台的原生界面能力，因此不能默认使用：

- HTML 标签
- 浏览器 DOM API
- CSS 的全部能力
- 仅存在于浏览器中的事件和布局机制

Expo UI 更进一步，让 React 代码能够使用基于 Jetpack Compose 或 SwiftUI 的原生组件。

### Jetpack Compose

Jetpack Compose 是 Android 的现代声明式 UI 工具包。

对于 React Web 开发者，可以把它理解为 Android 平台用于声明界面的组件体系。它在“根据状态描述界面”这一思想上与 React 相似，但组件、布局、样式和运行环境属于 Android 原生体系。

> **文档明确说明**：Expo UI 的 Jetpack Compose 组件用于构建 Android 原生界面。

### SwiftUI

SwiftUI 是 Apple 提供的声明式 UI 框架，可用于构建 iOS 等 Apple 平台的界面。

它同样采用声明式开发方式，但使用的是 Apple 原生组件和平台能力，而不是 HTML 元素。

> **文档明确说明**：Expo UI 的 SwiftUI 组件用于构建 iOS 原生界面。

### “原生组件”意味着什么

这里的“原生”不是指组件看起来像移动端控件，而是指其底层界面由平台原生 UI 技术构建：

- Android 使用 Jetpack Compose
- iOS 使用 SwiftUI

**基于文档内容推导**：由于不同平台使用不同的原生 UI 技术，各平台组件的 API 和行为不一定完全一致。开发时应先确定需要平台专用组件，还是需要统一的跨平台组件。

## Expo UI 的三类组件入口

原文按照平台和复用范围，将组件分为三类。

### Jetpack Compose

Jetpack Compose 组件用于构建 Android 原生界面。

适合以下情况：

- 功能只面向 Android
- 需要使用 Android 平台特有的 UI 能力
- 可以接受为 Android 编写平台专用界面

对应文档入口：

```text
/versions/latest/sdk/ui/jetpack-compose
```

当前总览文档没有列出具体组件、属性、事件或代码示例，需要继续查阅 Jetpack Compose 子文档。

### SwiftUI

SwiftUI 组件用于构建 iOS 原生界面。

适合以下情况：

- 功能只面向 iOS
- 需要使用 Apple 平台特有的 UI 能力
- 可以接受为 iOS 编写平台专用界面

对应文档入口：

```text
/versions/latest/sdk/ui/swift-ui
```

当前文档顶部同时将 `tvOS` 列为支持平台，但平台列表仅将 SwiftUI 描述为构建 iOS 界面，没有进一步说明 tvOS 的具体组件范围和差异。

> **限制**：不能仅根据这篇总览判断每个 SwiftUI 组件是否都支持 tvOS。

### Universal

Universal 提供从单一代码源运行在以下平台的跨平台组件：

- Android
- iOS
- Web

对应文档入口：

```text
/versions/latest/sdk/ui/universal
```

这类组件最接近 React Web 开发者熟悉的“编写一次，在多个目标环境复用”的模式。

**基于文档内容推导**：如果产品需要同时支持 Android、iOS 和 Web，可以优先检查 Universal 是否提供所需组件；只有在通用组件无法满足需求时，再考虑平台专用组件。

> 原文没有说明 Universal 与平台专用组件之间的功能差异、性能差异或 API 对应关系，因此不能假设 Universal 包含全部 Jetpack Compose 和 SwiftUI 组件。

## Drop-in replacements

Expo UI 还提供 Drop-in replacements，即针对流行 React Native 社区库的 API 兼容替代方案。

对应文档入口：

```text
/versions/latest/sdk/ui/drop-in-replacements
```

“API 兼容替代”通常意味着替代组件会尽量保持原库的调用接口，从而降低迁移成本。

不过，当前文档只提供了入口，没有说明：

- 支持替换哪些社区库
- API 兼容程度
- 安装和迁移步骤
- 是否存在行为差异
- 是否需要修改原生工程
- 是否支持渐进式迁移

这些内容应以 Drop-in replacements 子文档为准。

## 关键选择流程

根据当前文档提供的分类，可以按以下流程选择组件：

1. 确认目标平台。
2. 如果只需要 Android，查看 Jetpack Compose 组件。
3. 如果只需要 iOS，查看 SwiftUI 组件。
4. 如果需要 Android、iOS 和 Web 共用代码，优先查看 Universal 组件。
5. 如果正在使用 React Native 社区库并计划迁移，查看 Drop-in replacements。
6. 进入对应子文档，确认具体组件、API 和平台限制。

> **基于文档内容推导**：平台分类是使用 Expo UI 前最重要的决策，因为它决定了代码能否跨平台复用。

## 容易误解的地方

### Expo UI 不等于 React Native 的全部 UI 能力

Expo UI 的目标是提供典型应用常用的功能和组件，但原文没有声明它覆盖全部 React Native UI 场景。

因此，不能推断：

- 所有 React Native 核心组件都有 Expo UI 版本
- 所有原生控件都已经封装
- 所有 Android 和 iOS 组件具有相同能力

### “从 React 构建”不等于“使用 React DOM”

Expo UI 仍然使用 React 的组件和状态管理思想，但其平台专用组件最终对应 Jetpack Compose 或 SwiftUI，而不是 HTML DOM。

Web 项目中的以下经验不能直接照搬：

- 通过标签语义判断组件行为
- 使用 DOM 查询和操作 API
- 使用完整 CSS 选择器体系
- 假设不同平台拥有一致的事件对象
- 假设组件一定具有 Web 对应元素

### “Universal”才明确包含 Web

原文明确将 Universal 描述为同时运行在 Android、iOS 和 Web 的组件。

不要因为整个包支持多个平台，就推断 Jetpack Compose 或 SwiftUI 分类下的组件也能运行在 Web：

- Jetpack Compose 面向 Android
- SwiftUI 面向 iOS
- Universal 明确面向 Android、iOS 和 Web

### 包含在 Expo Go 不代表所有行为完全一致

原文明确标注 Expo UI 包含在 Expo Go 中，但没有说明：

- Expo Go 中包含哪些具体组件
- 是否存在版本对应要求
- 开发构建与 Expo Go 是否完全等价
- 各平台是否拥有相同的组件覆盖范围

这些问题需要结合具体组件文档确认。

## 实际开发建议

以下内容属于**基于经验建议**：

1. 在选型时先列出项目目标平台，再选择 Universal 或平台专用组件。
2. 不要看到 `@expo/ui` 包级别的平台列表后，就默认包内每个组件都支持所有平台。
3. 尽量把平台专用组件隔离在明确的模块中，避免平台判断散落在业务代码里。
4. 跨平台项目应先验证 Universal 组件是否满足需求，再引入 Jetpack Compose 或 SwiftUI 专用实现。
5. 使用 Drop-in replacement 前，应逐项核对原库与替代库的属性、事件、默认行为和平台支持情况。
6. 测试时应在每个目标平台实际运行。原生组件的外观和交互行为可能受到操作系统影响，不能只根据 Web 或单个平台结果判断。

## 文档信息边界

### 文档明确说明的内容

- Expo UI 的包名是 `@expo/ui`。
- 它提供一组原生输入组件。
- 它允许开发者从 React 使用 Jetpack Compose 和 SwiftUI 构建原生界面。
- 其目标是覆盖典型应用常用的功能和组件。
- Jetpack Compose 组件用于 Android。
- SwiftUI 组件用于 iOS。
- Universal 组件可通过单一代码源运行在 Android、iOS 和 Web。
- Expo UI 支持 Android、iOS 和 tvOS，并包含在 Expo Go 中。
- Expo UI 提供针对流行 React Native 社区库的 API 兼容替代方案。

### 基于文档内容推导的结论

- 跨平台项目可以优先评估 Universal 组件。
- 平台专用组件不应被默认视为支持 Web。
- 使用前需要先根据目标平台选择对应组件分类。
- 不同平台的原生组件可能存在 API 或行为差异，应在子文档中逐项确认。

## 总结

Expo UI 是 React 与现代原生 UI 技术之间的组件层：Android 使用 Jetpack Compose，iOS 使用 SwiftUI，Universal 组件则面向 Android、iOS 和 Web。

对 React Web 开发者而言，最重要的是不要把这些组件理解成另一套 HTML 组件。使用前应先确定目标平台，再进入对应分类查看具体 API、安装方式和限制。当前文档属于入口总览，负责说明 Expo UI 的定位和文档分类，并不提供完整的开发步骤。
