# Expo Router UI 学习文档

## 文档解决的问题

`expo-router/ui` 是 `expo-router` 提供的一个子模块，用于构建**自定义标签页布局（Custom Tab Layout）**。

它提供的是一组“无头组件（Headless Components）”和 Hooks：

- 负责标签页的路由状态、页面切换和事件处理。
- 不提供固定的标签栏外观。
- 开发者可以自行决定标签栏的结构、样式和交互方式。

它适合以下场景：

- 默认标签栏无法满足产品设计。
- 需要完全自定义标签按钮、布局或动画。
- 需要在标签栏以外的位置放置切换入口。
- 需要使用自定义组件代替 Expo Router 默认提供的 React Navigation 导航器 UI。

支持平台：

- Android
- iOS
- tvOS
- Web
- Expo Go

> 文档明确说明：`expo-router/ui` 是 `expo-router` 的子模块，不是一个需要单独安装的软件包。

---

## 阅读前需要理解的背景

### Expo Router

Expo Router 是一个同时面向原生应用和 Web 的**基于文件的路由系统**。

对于 React Web 开发者，可以将它类比为 Next.js 的文件路由：

- 文件和目录结构决定可访问的页面。
- 路由可以通过路径跳转。
- 布局文件负责组织一组页面的导航结构。

不同之处是，Expo Router 不只处理浏览器 URL，还需要管理 iOS、Android 等原生平台上的导航栈和屏幕生命周期。

### React Navigation

Expo Router 底层使用 React Navigation 管理原生导航。

默认情况下，Expo Router 会提供已经封装好的 Stack、Tabs 等导航器。`expo-router/ui` 则允许开发者绕过默认标签栏表现，自行组合标签页导航。

可以将两者关系理解为：

```text
React Navigation
  └── 提供底层原生导航状态和导航器能力
       └── Expo Router
            ├── 提供文件路由和默认导航器
            └── expo-router/ui
                 └── 提供自定义标签页布局所需的无头组件和 Hooks
```

### 无头组件

“无头”不是指组件不渲染任何内容，而是指组件主要提供行为和状态，不强制规定视觉样式。

这和 React Web 中的 Headless UI、Radix Primitives 等思路类似：

- 库负责状态和交互语义。
- 开发者负责最终 DOM 或原生视图结构及样式。

在 React Native 中，没有浏览器原生的 `<div>`、`<button>`。常见基础组件包括：

- `View`：类似布局容器。
- `Pressable`：可响应点击、长按等操作的组件。
- `Text`：用于显示文本。

## 安装与配置

### 安装

使用 `expo-router/ui` 前，项目必须已经安装并配置 `expo-router`。

当前文档没有提供完整安装命令，而是要求按照 Expo Router 的安装指南操作。因此不能仅根据本页确定项目所需的全部依赖和初始化步骤。

导入时使用子模块路径：

```tsx
import {
  Tabs,
  TabList,
  TabTrigger,
  TabSlot,
} from 'expo-router/ui';
```

不要将它理解成需要安装名为 `expo-router/ui` 的独立 npm 包。

### App Config 配置

Expo 应用使用 App Config 保存项目级配置，常见文件为：

```text
app.json
```

配置插件的示例：

```json
{
  "expo": {
    "plugins": ["expo-router"]
  }
}
```

这里的 `expo-router` 是 Expo Config Plugin。它用于在构建应用时处理 Expo Router 所需的项目配置。

如果项目通过 Expo 的默认模板创建，该插件通常已经配置好，不需要重复添加。

> 文档明确说明：默认模板已经配置 `expo-router` Config Plugin。

> 当前文档未说明该插件具体修改了哪些 iOS 或 Android 原生配置，也未提供手动修改原生工程的步骤。

## 组件模型

一个基础的自定义标签页布局由以下结构组成：

```tsx
<Tabs>
  <TabSlot />

  <TabList>
    <TabTrigger name="home" href="/" />
  </TabList>
</Tabs>
```

四个核心组件分别承担不同职责：

| 组件 | 职责 | React Web 类比 |
| --- | --- | --- |
| `Tabs` | 创建并维护标签页导航器 | 标签页状态 Provider 或路由布局根组件 |
| `TabSlot` | 渲染当前激活的标签页面 | React Router 的 `<Outlet />` |
| `TabList` | 声明标签页集合 | 标签栏容器 |
| `TabTrigger` | 声明或触发标签页切换 | 标签按钮或导航链接 |

这四个组件的结构不仅影响视觉布局，也会影响路由注册和导航行为。

---

## Tabs：标签导航根组件

`Tabs` 是无头标签导航器的根组件。

```tsx
<Tabs>
  <TabSlot />

  <TabList>
    <TabTrigger name="home" href="/" />
  </TabList>
</Tabs>
```

### `asChild`

```ts
asChild?: boolean
```

启用后，`Tabs` 会将自身属性转交给子组件，并移除默认生成的额外 `View`。

这适合需要自定义外层容器的场景，可以避免产生一层额外布局节点。

React Web 开发者可以将其类比为某些无头组件库中的 Slot 模式，但这里处理的是 React Native 的组件结构，而不只是浏览器 DOM。

使用时需要确保子组件能够正确接收转发的属性。

### `options`

```ts
options?: UseTabsOptions
```

用于向 Tab Router 传递导航器选项。其中本页明确列出的扩展选项是：

```ts
backBehavior?: TabRouterOptions['backBehavior']
```

`backBehavior` 用于控制执行“返回”操作时标签导航器的处理方式。

当前文档没有列出它支持的具体取值，应查阅 React Navigation 的 `TabRouterOptions` 文档，不能仅凭本页推断。

### 继承属性

`Tabs` 继承 React Native 的 `ViewProps`，因此可以接收 `View` 支持的布局、可访问性和事件等属性。

---

## TabList：声明标签页集合

`TabList` 是 `TabTrigger` 的包装组件。

```tsx
<TabList>
  <TabTrigger name="home" href="/" />
</TabList>
```

放在 `TabList` 内部的 `TabTrigger` 不只是一个切换按钮，还会定义 `Tabs` 中存在哪些标签路由。

这是它与普通 UI 按钮最重要的区别。

### `asChild`

```ts
asChild?: boolean
```

启用后：

- 属性会转交给子组件。
- 默认的额外 `View` 会被移除。
- 可以使用自定义组件作为标签列表容器。

### 继承属性

`TabList` 继承 React Native 的 `ViewProps`。

> 容易误解：文档中使用了 “TabTriggers” 作为复数描述，但实际导出的组件名是 `TabTrigger`。

---

## TabTrigger：声明或切换标签页

`TabTrigger` 用于创建标签页切换入口。

它的行为取决于是否位于 `TabList` 内部。

### 在 `TabList` 内部

```tsx
<TabList>
  <TabTrigger name="home" href="/" />
</TabList>
```

此时：

- `href` 是必需的。
- `name` 定义标签页名称。
- 该组件同时注册一个标签路由。

### 在 `TabList` 外部

位于 `TabList` 外部时：

- `href` 不再是必需的。
- `name` 用于引用已经定义的标签。
- 组件主要承担切换现有标签的作用。

因此，可以在自定义标签栏以外放置额外的标签切换入口。

### `name`

```ts
name: string
```

`name` 始终必填，但含义受使用位置影响：

| 位置 | 含义 |
| --- | --- |
| `TabList` 内部 | 设置标签页名称 |
| `TabList` 外部 | 引用已经存在的标签页名称 |

定义和引用时必须使用一致的名称。

### `href`

```ts
href?: Href
```

用于指定标签对应的 Expo Router 路由。

虽然类型声明中是可选属性，但在 `TabList` 内部使用时是必需的。这属于“类型层面可选、特定上下文中必填”的约束。

> 文档中的属性说明将 `href` 描述为 “Name of tab”，但结合其 `Href` 类型及上下文，它实际表示标签对应的路由目标。这里应以 API 类型和 `TabList` 的行为说明为准。

### `resetOnFocus`

```ts
resetOnFocus?: boolean
```

切换到该标签时重置路由状态。

这在每个标签内部拥有独立导航历史时尤其重要。例如：

```text
首页标签
  首页 → 商品列表 → 商品详情

设置标签
  设置首页
```

从设置标签重新点击首页标签时：

- 未重置：可能仍显示商品详情。
- 启用重置：回到该标签的初始路由。

文档中的 `SwitchToOptions` 进一步将其解释为：聚焦路由时执行导航并重置历史。

### `asChild`

```ts
asChild?: boolean
```

将触发器属性转交给自定义子组件，适合使用自定义标签按钮。

与 `Tabs`、`TabList` 的说明相比，这里只明确提到属性转发，没有明确写出会移除额外 `View`。

### 继承属性

`TabTrigger` 继承 `PressablePropsWithoutFunctionChildren`。

这意味着它具有 React Native `Pressable` 的大部分交互属性，但不支持函数形式的 `children`。

## TabSlot：渲染当前标签页面

`TabSlot` 用于渲染当前激活的标签页：

```tsx
<Tabs>
  <TabSlot />

  <TabList>
    <TabTrigger name="home" href="/" />
  </TabList>
</Tabs>
```

它可以类比为 React Router 的 `<Outlet />`：本身不是具体业务页面，而是当前匹配页面的渲染位置。

`TabSlot` 可以放在 `TabList` 前面或后面，从而决定内容区域和标签栏的布局顺序。

### `detachInactiveScreens`

```ts
detachInactiveScreens?: boolean
```

用于移除非激活页面。

其开发影响可能包括：

- 减少非激活页面占用的渲染资源。
- 非激活页面的视图可能不再保留在当前原生视图层级中。

> 文档明确说明的是 “Remove inactive screens”。本页没有详细解释移除后组件状态是否保留、何时重新挂载，以及各平台上的具体差异，因此不要将其直接等同于 React 的卸载。

### `renderFn`

```ts
renderFn?: defaultTabsSlotRender
```

用于覆盖 `Screen` 组件的渲染方式。

渲染函数会涉及以下状态：

| 字段 | 含义 |
| --- | --- |
| `index` | 页面在标签集合中的索引 |
| `isFocused` | 页面当前是否获得焦点 |
| `loaded` | 页面是否已经加载 |
| `detachInactiveScreens` | 非激活页面是否应从视图层级移除 |

该能力适用于需要深度控制屏幕容器和页面渲染方式的场景。

### 继承属性

`TabSlot` 继承 `ScreenContainer` 的组件属性。

## 组件与 Hook 的对应关系

`expo-router/ui` 同时提供组件式 API 和 Hook API：

| 组件能力 | Hook 形式 |
| --- | --- |
| `Tabs` | `useTabsWithChildren` 或 `useTabsWithTriggers` |
| `TabSlot` | `useTabSlot` |
| `TabTrigger` | `useTabTrigger` |

一般情况下优先使用组件 API。Hooks 适合需要封装自己的组件体系或直接读取导航状态的场景。

---

## useTabSlot

```tsx
function MyTabSlot() {
  const slot = useTabSlot();

  return slot;
}
```

也可以传入 `TabSlotProps`：

```ts
useTabSlot(namedParameters?: TabSlotProps): ReactElement
```

它返回当前标签页对应的 `ReactElement`，允许开发者在元素外部添加自定义结构：

```tsx
function MyTabSlot() {
  const slot = useTabSlot();

  return <CustomScreenContainer>{slot}</CustomScreenContainer>;
}
```

文档在 Components 和 Hooks 两个位置都列出了 `useTabSlot`，实际指向的是同一个 Hook 能力。

---

## useTabsWithChildren

```ts
useTabsWithChildren(options: UseTabsWithChildrenOptions)
```

它是 `Tabs` 的 Hook 版本，通过 `children` 解析标签触发器。

```tsx
export function MyTabs({ children }) {
  const { NavigationContent } = useTabsWithChildren({ children });

  return <NavigationContent />;
}
```

调用后必须渲染返回的 `NavigationContent`，否则导航内容不会进入组件树。

使用这个 Hook 时，仍需使用 Expo Router 导出的：

- `TabList`
- `TabTrigger`

返回结果还包含：

- `state`：标签导航器状态。
- `navigation`：导航操作对象。
- `descriptors`：路由描述信息。
- `NavigationContent`：必须渲染的导航内容组件。

文档展示的完整返回类型非常复杂，主要来自 React Navigation 的泛型组合。普通使用场景不需要手动重写这段返回类型，通常让 TypeScript 自动推断即可。

---

## useTabsWithTriggers

```ts
useTabsWithTriggers(options: UseTabsWithTriggersOptions)
```

这是另一种 `Tabs` Hook 形式。它不从 `children` 中发现触发器，而是通过 `triggers` 显式传入标签定义：

```ts
type UseTabsWithTriggersOptions = UseTabsOptions & {
  triggers: ScreenTrigger[];
};
```

适合以下情况：

- 标签列表来自配置数据。
- 需要实现完全自定义的标签组件体系。
- 不希望使用 `TabList` 和 `TabTrigger` 作为标签声明来源。

> 原文示例存在明显不一致：`useTabsWithTriggers` 小节中的代码调用了 `useTabsWithChildren({ triggers: [] })`。根据小节说明和参数类型，这里预期应调用 `useTabsWithTriggers`。本文不擅自认定实际 API 行为，开发时应结合对应 Expo SDK 版本的类型声明或源码确认。

---

## useTabTrigger

```ts
useTabTrigger(options: TabTriggerProps): UseTabTriggerResult
```

该 Hook 用于创建完全自定义的 `TabTrigger`。

返回值包括：

| 字段 | 作用 |
| --- | --- |
| `trigger` | 当前触发器信息，可能不存在 |
| `triggerProps` | 应绑定到自定义可交互组件上的属性 |
| `getTrigger(name)` | 按名称查找触发器 |
| `switchTab(name, options)` | 切换到指定标签 |

切换时可传入：

```ts
type SwitchToOptions = {
  resetOnFocus?: boolean;
};
```

典型实现思路：

```tsx
function CustomTabButton(props) {
  const { triggerProps } = useTabTrigger(props);

  return <Pressable {...triggerProps}>{/* 自定义内容 */}</Pressable>;
}
```

以上代码用于解释组合方式；原文没有提供完整的 `useTabTrigger` 示例。

## 导航事件

`TabNavigationEventMap` 定义了两类标签事件：

| 事件 | 触发时机 | 特点 |
| --- | --- | --- |
| `tabPress` | 点击标签时 | 可以阻止默认行为 |
| `tabLongPress` | 长按标签时 | 事件数据为 `undefined` |

`tabPress` 带有：

```ts
canPreventDefault: true
```

这意味着事件监听器可以阻止默认的标签切换行为。

对 React Web 开发者来说，可以将其类比为调用 `event.preventDefault()` 阻止链接默认跳转，但这里处理的是 React Navigation 事件，不是浏览器 DOM 事件。

## 屏幕生命周期选项

`ExpoTabsNavigatorScreenOptions` 包含以下可选属性：

| 属性 | 文档给出的类型 | 主要关注点 |
| --- | --- | --- |
| `detachInactiveScreens` | `boolean` | 是否移除非激活屏幕 |
| `freezeOnBlur` | `boolean` | 页面失去焦点后是否冻结 |
| `lazy` | `boolean` | 是否延迟加载页面 |
| `unmountOnBlur` | `boolean` | 页面失去焦点后是否卸载 |

原文只列出了字段，没有进一步说明具体行为和默认值。

这些选项不能视为同一个概念：

- `lazy` 关注页面第一次何时加载。
- `freezeOnBlur` 关注页面失焦后是否继续更新。
- `unmountOnBlur` 关注页面失焦后是否卸载。
- `detachInactiveScreens` 关注非激活页面是否保留在原生视图层级。

> 基于文档内容推导：这些选项会影响性能、组件状态保留以及重新进入标签时的表现，应在确认目标 Expo 和 React Navigation 版本的准确语义后再配置。

## 其他公开类型

### `ExpoTabsScreenOptions`

它从 React Navigation 的 `BottomTabNavigationOptions` 中选择了：

- `title`
- `lazy`
- `freezeOnBlur`

并扩展：

| 属性 | 类型 |
| --- | --- |
| `action` | `NavigationAction` |
| `params` | `object`，可选 |
| `title` | `string` |

这说明 Expo Router 的标签页面描述不仅包含显示信息，还可能携带导航动作和路由参数。

### `TabTriggerOptions`

```ts
type TabTriggerOptions = {
  href: Href;
  name: string;
};
```

表示一个基础标签定义需要名称和路由目标。

### `Trigger`

触发器解析后的信息包括：

| 属性 | 含义 |
| --- | --- |
| `isFocused` | 标签当前是否激活 |
| `resolvedHref` | 解析后的最终路由字符串 |
| `route` | 对应的路由信息 |

原文将 `route` 的类型显示为 `[number]`，但没有解释其数据结构。该类型展示可能不完整，不应根据本页自行构造该字段。

```ts
type TabsContextValue = ReturnType<typeof useNavigationBuilder>;
```

它表示由 React Navigation `useNavigationBuilder` 创建的自定义导航器上下文。

这是偏底层的扩展接口。只使用 `Tabs`、`TabList`、`TabTrigger` 和 `TabSlot` 时，通常不需要直接操作它。

### `TabContext`

`TabContext` 是标签导航上下文相关的 React 元素，使用 `ExpoTabsNavigatorScreenOptions`。

当前文档只提供类型，没有给出直接使用方式或示例。

### 导航器类型

以下类型主要用于高级封装和 TypeScript 类型约束：

- `ExpoTabsNavigationProp`
- `ExpoTabsNavigatorOptions`
- `UseTabsOptions`
- `UseTabsWithChildrenOptions`
- `UseTabsWithTriggersOptions`
- `UseTabTriggerResult`

它们大量组合了 React Navigation 的泛型。普通业务代码应优先依赖 API 的类型推断，只有在封装导航组件或公共库时才需要显式引用这些类型。

## React Web 开发者容易误解的地方

### 标签切换不只是条件渲染

在 React Web 中，简单标签页经常写成：

```tsx
{activeTab === 'home' && <Home />}
```

`expo-router/ui` 的标签页不是单纯的条件渲染。它还涉及：

- 文件路由匹配。
- 标签导航状态。
- 每个标签内部的导航历史。
- 页面焦点和失焦。
- 屏幕挂载、冻结及移除策略。
- 原生返回操作。

因此，不应自行用一个 `useState` 替换其导航逻辑。

### `href` 和 `name` 不是同一个标识

- `name` 是标签导航器内部使用的标签名称。
- `href` 是 Expo Router 的路由目标。

它们可能看起来相似，但承担不同职责。

### “失去焦点”不等于浏览器窗口失焦

这里的 blur/focus 指导航页面是否为当前活动页面，不是浏览器的：

```js
window.addEventListener('blur', ...)
```

也不是输入框的焦点状态。

### 隐藏、移除、冻结和卸载不同

移动端导航通常会保留多个屏幕，以便恢复状态和实现流畅切换。

因此：

- 页面不可见不代表已卸载。
- 从原生视图层级移除不一定等于 React 组件卸载。
- 页面被冻结不代表状态被清空。
- 切换标签不一定回到标签首页。

应根据具体生命周期选项判断页面行为。

### `asChild` 会改变组件组合结构

使用 `asChild` 后，子组件需要能够接收被转发的属性和交互处理器。若自定义组件没有继续传递这些属性，可能导致：

- 点击无效。
- 可访问性属性丢失。
- 布局属性丢失。
- 导航状态无法正确反映到 UI。

---

## 注意事项与限制

1. `TabTrigger` 在 `TabList` 内时必须提供 `href`，即使 TypeScript 属性声明显示为可选。
2. `TabTrigger` 在 `TabList` 内用于定义标签，在外部用于引用标签，两种用法不要混淆。
3. 使用 `useTabsWithChildren` 后必须渲染返回的 `NavigationContent`。
4. `useTabsWithChildren` 仍依赖 Expo Router 提供的 `TabList` 和 `TabTrigger`。
5. 完全自定义触发器体系时，应考虑 `useTabsWithTriggers` 和 `useTabTrigger`。
6. `resetOnFocus` 会影响标签内部导航历史，不能只把它当作刷新页面的选项。
7. 生命周期相关属性会影响状态保留和性能，但本页没有给出默认值和完整语义。
8. API 返回类型中的部分内容在原文页面上显示不完整或格式异常，应以项目实际安装版本的 TypeScript 声明为准。
9. `useTabsWithTriggers` 的原文示例与该小节 API 名称不一致，直接照抄可能得到错误实现。
10. 本页是 API 参考，不包含完整的自定义标签页教程。实际布局步骤需要结合 Expo 的 Custom tab layouts 指南。

## 实际开发中的使用方式

### 优先采用基础组件组合

大多数自定义标签栏可以从以下结构开始：

```tsx
<Tabs>
  <TabSlot />

  <TabList>
    <TabTrigger name="home" href="/" />
    <TabTrigger name="settings" href="/settings" />
  </TabList>
</Tabs>
```

然后通过 React Native 样式、自定义容器以及 `asChild` 调整视觉表现。

### 仅在确有需要时使用底层 Hooks

建议按复杂度选择 API：

| 需求 | 建议 API |
| --- | --- |
| 自定义标签栏样式 | `Tabs`、`TabList`、`TabTrigger`、`TabSlot` |
| 自定义当前页面容器 | `useTabSlot` 或 `TabSlot.renderFn` |
| 封装自己的 Tabs 根组件 | `useTabsWithChildren` |
| 标签由配置动态声明 | `useTabsWithTriggers` |
| 完全自定义标签按钮行为 | `useTabTrigger` |

### 谨慎配置页面生命周期

基于经验建议：

在没有明确性能问题或产品需求前，不要同时随意启用 `detachInactiveScreens`、`freezeOnBlur` 和 `unmountOnBlur`。先确认以下需求：

- 切换标签后表单内容是否需要保留。
- 返回标签时滚动位置是否需要保留。
- 页面中的请求或定时器是否应继续运行。
- 每次进入标签是否需要重新初始化。
- 标签内部的导航历史是否需要保存。

### 使用实际版本的类型定义校验 API

基于经验建议：

由于本页存在示例和类型展示不完整的问题，实现高级 Hooks 前应查看编辑器中当前安装版本的类型提示。Expo SDK 版本和 `expo-router` 版本应保持兼容，不要直接依据其他版本文档复制底层类型。

## 文档明确说明与推导内容

### 文档明确说明

- `expo-router/ui` 用于构建自定义标签页布局。
- 它属于 `expo-router` 子模块。
- 它支持 Android、iOS、tvOS 和 Web，并包含在 Expo Go 中。
- 默认 Expo 项目模板已经配置 `expo-router` Config Plugin。
- `Tabs` 是无头标签导航的根组件。
- `TabList` 内的 `TabTrigger` 会定义标签路由。
- `TabTrigger` 在 `TabList` 内必须具有 `href`。
- `TabTrigger` 在 `TabList` 外不要求提供 `href`。
- `TabSlot` 渲染当前标签页。
- 组件能力分别有对应的 Hook API。
- `tabPress` 事件可以阻止默认行为。
- `resetOnFocus` 可以在切换标签时重置路由历史。

### 基于文档内容推导

- `TabSlot` 的角色类似 React Router 的 `<Outlet />`。
- `name` 是标签内部标识，`href` 是文件路由目标。
- 生命周期选项会影响性能、状态保留和重新进入页面时的行为。
- 常规业务场景优先使用组件 API，高级组件封装再使用 Hooks。
- `useTabsWithTriggers` 小节的示例可能误用了 `useTabsWithChildren`。
- 原文显示异常的复杂类型不适合作为手写类型，应依赖当前版本的 TypeScript 声明。

## 总结

`expo-router/ui` 将标签页导航拆分为四个核心职责：

```text
Tabs       → 建立标签导航器
TabList    → 声明标签集合
TabTrigger → 定义或切换标签
TabSlot    → 渲染当前标签页面
```

它适用于需要自定义标签栏外观和结构，同时仍希望保留 Expo Router 文件路由与 React Navigation 导航能力的项目。

对 React Web 开发者而言，最关键的是不要把它当成普通的前端 Tab 条件渲染。它管理的是跨 Web 和原生平台的路由、页面焦点、导航历史及屏幕生命周期。基础组件组合足以覆盖多数场景；Hooks 和复杂导航类型主要用于构建更底层的自定义导航组件。
