# Expo UI 概览

## 文档解决的问题

本文介绍 `@expo/ui` 的定位以及它提供的三类组件入口，帮助开发者选择合适的原生 UI 开发方式。

`@expo/ui` 允许开发者通过 React 使用以下原生 UI 技术构建界面：

- Android 的 Jetpack Compose
- iOS 和 tvOS 的 SwiftUI
- 同时支持 Android、iOS 和 Web 的 Universal 组件

本文属于概览和导航页面，并未深入介绍具体组件 API。

## 阅读前需要理解的背景知识

### 什么是 Expo

Expo 是围绕 React Native 提供的一套开发工具和服务。它简化了 React Native 项目的创建、调试、构建以及原生能力接入。

本文介绍的 `@expo/ui` 是 Expo SDK 中的一个 UI 组件包。

### React Native 与 React Web 的区别

React Web 最终通常渲染为浏览器 DOM，例如：

```jsx
<button>Submit</button>
```

React Native 不使用 HTML DOM，而是将 React 组件映射到移动平台的原生视图。

`@expo/ui` 更进一步：它提供的组件由现代原生 UI 框架实现，使 React 代码可以使用：

- Android 上的 Jetpack Compose 组件
- Apple 平台上的 SwiftUI 组件

因此，这里的“原生界面”不是指在 WebView 中显示网页，而是使用平台自身的 UI 系统渲染界面。

### Jetpack Compose

Jetpack Compose 是 Android 的声明式 UI 框架。

对于 React 开发者，可以将其大致理解为 Android 平台上的声明式组件系统：开发者描述当前状态对应的界面，框架负责更新实际显示内容。

### SwiftUI

SwiftUI 是 Apple 的声明式 UI 框架，主要用于构建 iOS、macOS、watchOS 和 tvOS 界面。

本文中的 `@expo/ui` 使用 SwiftUI 构建 iOS 和 tvOS 原生组件。

需要注意，Jetpack Compose 和 SwiftUI 是两套不同的平台技术，并不是 React Native 组件的两个名称。

## `@expo/ui` 的定位

文档将 `@expo/ui` 定义为一组原生输入组件，目标是覆盖典型应用经常需要的功能和组件。

包名为：

```text
@expo/ui
```

源码位于：

```text
https://github.com/expo/expo/tree/sdk-56/packages/expo-ui
```

文档列出的支持环境包括：

- Android
- iOS
- tvOS
- Expo Go

其中，Expo Go 是 Expo 提供的通用开发客户端，可以加载和预览受支持的 Expo 项目功能。本文明确标注 `@expo/ui` 包含在 Expo Go 中。

> **文档明确说明：** `@expo/ui` 的目标是提供典型应用常用的功能和组件。本文没有承诺它能够覆盖所有 Jetpack Compose 或 SwiftUI API。

## 三类组件入口

### Jetpack Compose 组件

Jetpack Compose 入口用于构建 Android 原生界面。

适合以下场景：

- 只需要实现 Android 界面
- 希望使用 Android 平台的原生组件
- 需要的平台能力在 Universal 组件中不存在

对于 React Web 开发者，这类似于选择一个仅面向 Android 的组件库。使用这类组件意味着界面代码具有平台针对性，不能仅凭同一实现直接推断其可运行于 iOS 或 Web。

具体组件和 API 需要继续阅读 Jetpack Compose 子文档。当前页面未提供导入方式、属性、事件或代码示例。

### SwiftUI 组件

SwiftUI 入口用于构建 iOS 原生界面，其页面元数据显示 `@expo/ui` 也支持 tvOS。

适合以下场景：

- 只需要实现 Apple 平台界面
- 希望使用 SwiftUI 提供的原生组件
- 需要的平台能力在 Universal 组件中不存在

SwiftUI 组件不等同于 Android 上的 Jetpack Compose 组件。即使两个平台存在外观或功能相近的控件，它们仍然属于不同的实现和 API 入口。

具体用法需要阅读 SwiftUI 子文档。当前页面未列出组件属性、事件模型或平台差异。

### Universal 组件

Universal 组件允许使用同一份源码运行于：

- Android
- iOS
- Web

这类组件最接近 React Web 开发者熟悉的跨平台组件思路。

> **基于文档内容推导：** 如果目标是尽量复用 Android、iOS 和 Web 代码，应优先检查 Universal 组件是否能够满足需求；只有需要平台专属组件或能力时，再考虑 Jetpack Compose 或 SwiftUI 入口。

“Universal”表示组件提供跨平台实现，并不表示不同平台的视觉效果和行为必然完全一致。当前页面没有说明各平台的一致性边界，因此不能据此假定像素级一致。

## Drop-in replacements

文档还提供了 **Drop-in replacements** 入口，用于查找热门 React Native 社区库的 API 兼容替代方案。

“API-compatible replacement”可以理解为：替代组件会尽量保持与原社区库兼容的 API，从而降低替换成本。

它适合以下场景：

- 项目已经使用某个 React Native 社区 UI 库
- 希望迁移到 `@expo/ui` 提供的对应实现
- 希望减少迁移时对现有调用代码的修改

不过，当前页面没有列出：

- 支持替换哪些社区库
- API 兼容范围
- 是否完全兼容
- 迁移步骤
- 行为差异和平台限制

这些内容需要阅读 Drop-in replacements 专题文档，不能仅根据“API-compatible”认定替换后完全没有行为变化。

## 如何选择组件类型

根据本文提供的三个入口，可以使用下面的判断方式：

| 开发目标 | 建议首先查看 |
| --- | --- |
| 只构建 Android 原生界面 | Jetpack Compose |
| 只构建 iOS 或 tvOS 原生界面 | SwiftUI |
| 希望同一份源码支持 Android、iOS 和 Web | Universal |
| 替换现有 React Native 社区组件库 | Drop-in replacements |

> **基于文档内容推导：** 在跨平台项目中，组件选择应以 Universal 为起点，再针对无法满足的需求采用平台专属实现。这样有利于提高代码复用率，但本文没有提供混合使用三类组件的具体架构方法。

## React Web 开发者容易误解的地方

### 原生组件不是 DOM 组件

这些组件不会生成 `div`、`input` 或 `button` 等 HTML 元素。Jetpack Compose 和 SwiftUI 组件由对应平台的原生 UI 系统渲染。

因此，不能默认以下 Web 能力仍然适用：

- CSS 选择器和层叠规则
- DOM 查询与操作
- 浏览器事件对象
- HTML 表单行为
- 浏览器开发者工具中的元素检查方式

当前页面未说明 `@expo/ui` 的具体样式和事件 API，应以各组件子文档为准。

### React 是调用入口，不代表底层实现跨平台统一

`@expo/ui` 允许从 React 使用原生组件，但 Android 和 Apple 平台仍然使用不同的底层框架。

只有 Universal 组件被本文明确描述为可以通过同一份源码运行于 Android、iOS 和 Web。

### “Included in Expo Go”不等于所有原生库都能直接使用

本文只明确说明 `@expo/ui` 包含在 Expo Go 中。不能由此推导其他任意 React Native 原生库也包含在 Expo Go 中。

### API 兼容不代表实现完全相同

Drop-in replacement 的目标是降低替换成本，但当前页面没有承诺外观、行为、性能以及所有边界情况都与原库完全一致。正式迁移前仍应阅读对应替换文档并进行测试。

## 实际开发中的使用方式

1. 确认目标平台是 Android、iOS、tvOS，还是还需要支持 Web。
2. 跨平台需求优先检查 Universal 组件。
3. Android 专属需求查看 Jetpack Compose 组件。
4. iOS 或 tvOS 专属需求查看 SwiftUI 组件。
5. 如果正在替换 React Native 社区库，查看 Drop-in replacements 页面确认兼容范围。
6. 在具体子文档中核实安装方式、组件 API、平台限制和示例后再开始实现。

> **基于经验建议：** 即使使用 Universal 组件，也应分别在 Android、iOS 和 Web 上测试布局、交互和可访问性，不要只根据单个平台的运行结果判断跨平台效果。

## 文档范围与结论来源

### 文档明确说明

- `@expo/ui` 是一组原生输入组件。
- React 可以通过它使用 Jetpack Compose 和 SwiftUI 构建原生界面。
- 它旨在提供典型应用常用的功能和组件。
- Jetpack Compose 入口面向 Android。
- SwiftUI 入口面向 iOS。
- Universal 组件支持 Android、iOS 和 Web，并可使用同一份源码。
- Drop-in replacements 提供热门 React Native 社区库的 API 兼容替代方案。
- 页面标注支持 Android、iOS、tvOS，并包含在 Expo Go 中。

### 基于文档内容推导

- 跨平台项目可优先评估 Universal 组件，再按需采用平台专属组件。
- 选择 Jetpack Compose 或 SwiftUI 会引入平台针对性的 UI 代码。
- Drop-in replacements 可以降低迁移成本，但迁移前仍需核实具体兼容范围。

### 当前文档未涉及

- 安装和初始化流程
- 完整组件列表

- Drop-in replacement 的具体支持列表和兼容边界

## 总结

`@expo/ui` 为 React 提供了访问现代原生 UI 组件的入口。开发者可以根据平台范围选择 Android 的 Jetpack Compose、Apple 平台的 SwiftUI，或者支持 Android、iOS 和 Web 的 Universal 组件。

本文只是功能概览和文档导航，主要帮助开发者确定后续应阅读哪个分支。真正接入项目时，还需要查阅对应子文档中的安装方法、组件 API 和平台限制。
