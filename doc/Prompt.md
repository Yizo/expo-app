# 技术文档阅读理解与中文 Markdown 生成

## 任务目标

请阅读我提供的技术文档，并基于原文档内容生成一份中文 Markdown 学习文档。

这不是简单摘要，而是要求你在完整理解原文档的基础上，帮助我系统掌握文档中的知识点、流程、注意事项、限制条件和实际开发影响.

## 文档导航（必须输出）

本篇文档在本地学习系列中的位置由工作流提供，请在**正文全部写完后**，在文档最末尾追加以下导航区块。

将占位符替换为工作流传入的最终内容；不要保留 `{{prev_nav}}` / `{{next_nav}}` 字样：

```md
---

## 文档导航

- **上一页**：{{prev_nav}}
- **下一页**：{{next_nav}}
```

要求：

1. 导航区块必须放在整篇文档最后。
2. 不要修改上一页、下一页的链接目标，只负责按给定内容原样输出。
3. 若占位符值为 `无`，则直接输出 `无`。

## 阅读和整理原则

请严格基于当前文档内容进行整理。

你需要先理解原文档的结构和意图，再决定最终 Markdown 文档的组织方式。

不要强行套用固定模板。

如果原文档是教程型文档，就按教程流程整理。

如果原文档是概念型文档，就按概念关系整理。

如果原文档是 API / 配置说明文档，就按配置项、参数、用法和注意事项整理。

如果原文档是故障排查或迁移文档，就按问题、原因、解决方案和风险整理。

最终输出结构应当服务于当前文档，而不是套用固定目录。

## 输出要求

请使用中文输出，并生成 Markdown 格式文档。

**工作流说明（不要写入文档正文）：** 若当前任务对应的目标文件 `{{行序号}}__{{文件名}}.md` 在输出目录中已存在，工作流会跳过生成，不会调用本提示词。以下要求仅适用于实际执行生成的任务。

输出时请遵守以下要求：

- 只输出单篇学习文档正文，不要输出本次生成概览、任务状态、跳过说明等工作流元信息
- 保留原文档的主要逻辑顺序
- 不要遗漏原文档中的重要知识点
- 不要遗漏注意事项、限制条件和警告
- 不要只翻译，要解释含义和开发影响
- 不要默认我懂 React Native 或 Expo
- 对关键术语要补充解释
- 示例代码必须要同步到新文档中,不得遗漏与删除
- 对命令和配置要说明作用，而不是只列出来
- 对容易误解的地方要重点说明
- 对文档没有明确说明的内容，不要猜测
- 如果给出经验性建议，请标注“基于经验建议”
- 如果是根据文档内容推导出来的结论，请标注“基于文档内容推导”
- 在文档顶部需要展示原文地址

## 推荐输出方式

请根据文档实际内容，自行设计 Markdown 结构。

但建议至少包含以下几类信息：

```md
# 文档标题

## 文档解决的问题

## 阅读前需要理解的背景知识

## 按原文结构整理的核心内容

## 关键概念解释

## 关键流程 / 命令 / 配置说明

## 注意事项、限制条件和坑点

## 总结

（正文结束后，再追加「文档导航」区块，见上文要求）
```

注意：以上只是推荐结构，不是强制模板。
如果当前文档不适合这个结构，请按照原文档本身的结构重新组织。

## 重点要求

请优先保证：

1. 忠实于原文档
2. 不遗漏重要信息
3. 对初学 React Native / Expo 的 React Web 开发者友好
4. 结构根据文档内容自然生成
5. 注意事项和限制条件必须重点提取

## 最终质量审查：去重与合并

在生成最终 Markdown 文档之前，必须对全文进行一次内容审查，重点检查是否存在重复性描述。

审查要求如下：

1. 检查不同章节中是否反复表达了相同含义。
2. 如果多个段落表达的是同一个知识点，应当合并为一处。
3. 如果某个注意事项、限制条件或坑点在多个位置都出现，应保留最合适的位置，其他位置删除或改为引用说明。
4. 对必须重复强调的重要风险，可以保留，但需要换成不同角度说明，不能简单复制同一句话。
5. 最终输出前，应确保文档结构清晰、表达紧凑、没有冗余段落。
6. 不要为了满足目录完整性而重复解释同一个概念。
7. 如果某个内容已经在前文解释过，后文只需要补充其开发影响、限制条件或使用场景，不要再次完整解释。

最终输出的 Markdown 文档应当是经过清理后的版本，而不是先生成一份重复内容较多的文档再说明“可能有重复”。
